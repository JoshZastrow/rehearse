# Product Requirements Document: RealTalk

**Version:** 1.0
**Date:** 2026-04-11
**Status:** Draft

## 1. Overview

### 1.0 Product Vision Defined

RealTalk is a terminal-based conversational game that simulates emotionally charged social interactions between a human player and an LLM-powered character. Players practice reading social dynamics, choosing contextually appropriate responses, and navigating the emotional arc of a relationship toward a moment of genuine connection.

The game is not about manipulation — it is about attunement. The player must track both their own internal state and the other character's emotional state, and make choices that reflect emotional intelligence.

### 1.1 Product Goals

- Deliver a compelling, replayable TUI game launched with a single command: `realtalk`
- Keep sessions focused (10–20 turns) and completable in under 15 minutes
- Use an LLM provider presented as a living, reactive character that creates conversational paths
    0. (First turn only): Human player enters a greeting and conversational opener.
    1. From then on, after each human input, LLM responds like a human woman would
    2. LLM generates conversation outputs reflecting mood shifts that are a result of previous player choices
    3. Player chooses from that set of prompts and gets scored for their choice
    4. LLM adjusts next comment to output to human
    REPEAT: STEPS 1-4 10-20 times and stop according to
Goal:
- Teach players (men especially) to recognize and respond to emotional and relational cues

### Session Length

- Target: 10–20 turns per session
- Minimum turns to win: 8 (prevents trivial play)
- Turn hard cap: 25 (if not resolved, the character "disengages" — neutral loss)

### 1.1 Parameters that equal winning

### Scoring & Post-Game Summary

Displayed at end of game. Components:

| Component | Description |
|-----------|-------------|
| Consistency | Did the player's self-ratings (a/r, intensity) track coherently with their choices? |
| Attunement | How well did player choices correlate with positive mood/security shifts? |
| Patience | Bonus for reaching the win condition without rapid mood swings (volatility penalty) |
| Final Arc | Win/loss on the Invitation Arc (binary) |

### 1.1 Out of Scope (v1)

- Post-game summary
- Player rates their internal reaction
- Multiplayer or networked play
- Persistent save/load between sessions
- Character memory across multiple game sessions
- Mobile or web interface
- Any explicit or graphic content generation
- Crosss session-support
- Role selection / personality choice for who the player wants to talk to

## High-Level Flow Game Flow
```
Start
  └── Scene Selection (3 options)
        └── Role Selection (5 options)
              └── Situation Description (LLM-generated)
                    └── LLM Character Opening Line
                          └── [Game Loop]
                                ├── Player Rates Own Reaction (direction + intensity)
                                ├── Player Chooses Response (3 LLM-generated options)
                                ├── LLM Character Responds
                                ├── LLM Updates Mood Bar + Security Bar
                                └── Check Win Condition
                                      ├── Mood < 80 → repeat loop
                                      └── Mood ≥ 80 → Invitation Arc (2–4 turns)
                                              ├── Right Response → WIN
                                              └── Wrong Response → LOSE
```

## Onboarding - set roles, scene and description

### Scene Selection

At game start, the player is presented with three scenes, randomly drawn from the scene pool below. Scenes establish physical setting, social context, and the nature of the interaction.

**Scene Pool (minimum 6 at launch):**

| Scene ID | Name | Description |
|----------|------|-------------|
| S-01 | Coffee Shop | You've both arrived at the same small café table. It's raining. |
| S-02 | Late Night Office | You're both still at the office after hours, finishing a project. |
| S-03 | Hiking Trail | You've just reached a ridge together after a long climb. |
| S-04 | House Party | You've ended up in the same quiet corner of a loud party. |
| S-05 | Bookstore | You keep running into each other in the same aisle. |
| S-06 | Airport Gate | A delayed flight, two seats together at the gate. |

**Example Scene Output (Coffee Shop + Friend) and first prompt:**

> *The rain started about twenty minutes ago, and neither of you planned for it. Your usual table by the window is free, and you've both claimed it out of habit. The café smells like cardamom and wet coats.*
>
> *She's drumming her fingers on the table, not quite looking at you.*
>
> **"You always pick the window seat. Don't you think it gets drafty?"**

> {LLM then responds according to their role}

### Role Selection

After scene selection, the player chooses the relationship role the LLM character will play. This defines the character's baseline personality, communication style, and win-condition tone.

| Role | Key Traits | Win-Condition Tone |
|------|-----------|-------------------|
| Girlfriend | Warm, emotionally expressive, mildly testing | Direct, tender, personal |
| Friend | Playful, guarded, slow to open up | Casual but vulnerable |
| Co-worker | Professional exterior, curious undercurrent | Ambiguous, boundary-aware |
| Teammate | Competitive, high-trust, direct | Challenge wrapped in warmth |
| Teacher | Composed, observational, expects depth | Intellectual + personal blending |

## Technical Requirements

### 1. CLI Entry Point

- Command: `realtalk`
- Installed globally via npm/pip/cargo/homebrew (TBD by implementation language)
- No flags required for standard play
- Optional flags for v1:
  - `--debug` — Show LLM prompt structure alongside game output
  - `--no-color` — Disable color/styling for accessibility
  - `--contributor` — Enable Contributor Mode (see §10.6)

### 2. TUI Framework

The interface must support:
- Full-screen terminal rendering
- Color-coded bars (green for mood, blue for security)
- Text regions: dialogue area, status area, input area
- Responsive to common terminal widths (minimum 80 columns)
- Graceful exit on `Ctrl+C` with prompt ("Quit game? [y/n]")

Recommended frameworks (by language):
- **Node.js:** `blessed`, `ink`, or `terminal-kit`
- **Python:** `textual` or `urwid`
- **Rust:** `ratatui`
- **Go:** `bubbletea`

### 3. LLM Integration

| Concern | Requirement |
|---------|-------------|
| Provider | Anthropic Claude (claude-haiku-4-5 for fast turn responses) |
| API key | Read from `ANTHROPIC_API_KEY` environment variable |
| Prompting | System prompt encodes scene, role, character personality, and current game state |
| State passing | Full turn history passed each call (within context window) |
| Response format | Structured JSON: `{ dialogue: string, mood_delta: { direction, intensity }, security_delta: { direction, intensity }, invite_turn: boolean }` |
| Latency | Target < 3s per LLM response; show a spinner while waiting |
| Error handling | Retry once on failure; surface error and offer to continue or quit |

### 4. Response Option Generation

Three response options are generated per turn via a separate LLM call (or as part of the character-response call, structured output). Options must be:
- Contextually plausible within the scene and role
- Meaningfully differentiated (open / deflect / withdraw)
- Under 25 words each
- Written in first person

### 5. Configuration

At launch, if `ANTHROPIC_API_KEY` is not set, prompt the user:
```
No API key found. Set ANTHROPIC_API_KEY in your environment to play.
export ANTHROPIC_API_KEY=sk-...
```

### 6. Contributor Mode

Contributor Mode is an opt-in data collection feature for RLHF and model training. It is **never enabled by default.**

**Activation:** `realtalk --contributor`

On first use, the player is shown a consent screen explaining:
- What data is collected (turn sequence, player reaction ratings, response choices, bar deltas, win/loss outcome)
- That data is anonymized before any submission
- That they can opt out at any time by omitting the flag

**What is collected per turn:**
```json
{
  "turn": 3,
  "character_message": "...",
  "player_reaction": { "direction": "a", "intensity": 2 },
  "response_options": ["...", "...", "..."],
  "player_choice": 1,
  "mood_delta_player": { "direction": "a", "delta": 7 },
  "mood_delta_character": { "direction": "a", "delta": 5 },
  "security_delta": { "direction": "c", "delta": 3 },
  "arc_active": false
}
```

**Storage:** Written to a local JSONL file (`~/.realtalk/sessions/`). Submission to a remote endpoint is a future capability — v1 only writes locally.

**Privacy guarantee:** Player reaction ratings are **never** included in the active LLM prompt under any mode. The contributor flag only affects local storage, not runtime behavior.

# FUTURE: EXPANDED FEATURES

## Invitation Arc (Endgame)

### Trigger

When the Mood bar reaches **80 or above**, the Invitation Arc begins. The threshold is 80, not 100 — the bar does not need to be maxed. This means a player can sustain attraction without perfection, and the arc can begin while there is still room to gain or lose ground.

The character does **not** immediately issue an invitation — instead, their dialogue shifts tonally over 2–4 turns, building toward a moment of genuine opening. This shift should be readable but not explicit.

If mood drops back below 80 during the arc (e.g., the player rates `r3` or makes a poor response), the arc is suspended and the game returns to the standard loop. The arc resumes when mood crosses 80 again.

**Example signals:**
- They reference something personal they haven't mentioned before
- They slow the pace of their sentences
- They ask a question that invites the player to be real
- A pause is described in the scene-setter prose

### The Invitation

On the designated Invitation Turn, the character says something that represents a genuine, role-appropriate invitation for deeper connection. The nature of the invitation depends on role:

| Role | Invitation Type |
|------|----------------|
| Girlfriend | Personal, emotionally direct — "I've been thinking about you." |
| Friend | Vulnerability offered — "This is the most myself I've been in months." |
| Co-worker | Boundary shift — "This is probably unprofessional, but..." |
| Teammate | Trust marker — "I don't say this to everyone. You're good at this." |
| Teacher | Recognition — "You ask the questions I want to be asked." |

### Win/Lose Determination

The player is shown three responses as usual. One is genuinely attuned — it meets the character's emotional offer with presence, not performance. The others are:
- **Deflecting too hard** (kills the moment)
- **Overclaiming** (pushes too fast, breaks trust)

The LLM evaluates the player's choice against the character's current mood, security state, role personality, and the cumulative turn history.

- **Right choice:** WIN — the character's final message signals genuine connection. A score summary is shown.
- **Wrong choice:** LOSE — the character visibly retreats. A brief note on what went wrong is shown. Option to replay.

### Player Reaction Rating

After reading the character's message, before choosing a response, the player logs their internal reaction. This functions like a Street Fighter health bar — it directly impacts the character's Mood bar. An `a` rating charges the bar; an `r` rating drains it. The intensity determines the magnitude of the hit.

**Direction:** Single keypress
- `a` — Attraction (drawn toward, engaged, curious, warmer) → **raises** Mood bar
- `r` — Repulsion (pulled back, guarded, uncomfortable, cooler) → **lowers** Mood bar

**Intensity:** Single keypress
- `1` — Low (subtle, mild) → ±3 mood points
- `2` — Medium (noticeable, real) → ±7 mood points
- `3` — High (strong, immediate) → ±12 mood points

The reaction rating is applied to the Mood bar immediately on submission, before the player picks a response. The response choice then triggers a second mood shift from the LLM character. Both shifts are visible in the bar update at the end of the turn.

The raw rating is **never sent to the LLM.** The character cannot "see" the player's internal state — only the chosen response. This preserves the authenticity of the interaction: a player can feel `r3` (strong repulsion) and still choose an open response, and the character responds only to what was said.

In **Contributor Mode** (see §10.6), ratings are stored locally and may be submitted for RLHF/model training purposes. This is always opt-in and anonymized.

**Prompt displayed:**
```
How are you feeling? [a/r] + [1/2/3]
> a2
```

### Response Selection

After rating, the player is shown three response options. These are LLM-generated, contextually appropriate, and meaningfully differentiated:

- **Option 1:** Emotionally open / vulnerable
- **Option 2:** Playful / deflecting
- **Option 3:** Withdrawn / practical / topic-changing

Options are not labeled by type — the player must read and choose. The game does not tell the player which is "correct."

**Display format:**
```
Choose your response:

  1. "I like the draft. Keeps me from getting too comfortable."
  2. "You've been tracking my seat preferences? Should I be flattered?"
  3. "I can move if it's really bothering you."

> _
```

## Fine-Grained Status Bars

### Mood Bar

Tracks the character's emotional engagement — their interest, warmth, and openness in the interaction.

| Property | Value |
|----------|-------|
| Range | 0–100 |
| Start | 30–50 (randomized per session, per role) |
| Arc Trigger | Reaches ≥ 80 (Invitation Arc begins) |
| Lose Trigger | Drops to 0 (character disengages) |
| Direction Label | `(a)ttraction` / `(r)epulsion` |
| Intensity | 1–3 per turn shift |
| Player reaction | `a` raises bar, `r` lowers bar (±3/7/12 pts by intensity) |

**Bar display:**
```
MOOD  [===========>         ] 58  (+a2)
```

### Security Bar

Tracks the character's sense of emotional safety and trust in the interaction. Distinct from mood — a character can be excited but insecure (high mood, low security) or calm and trusting (moderate mood, high security).

| Property | Value |
|----------|-------|
| Range | 0–100 |
| Start | 40–60 (randomized per session, per role) |
| Direction Labels | `(c)omfort` / `(p)rotective` |
| Intensity | 1–3 per turn shift |

- **Comfort (c):** Moving toward openness, trust, ease
- **Protective (p):** Moving toward guardedness, withdrawal, self-protection

Security does not have a win/lose threshold of its own, but it strongly influences how the character interprets player responses and which responses qualify as "right" during the Invitation Arc.

**Bar display:**
```
SECURITY  [============>        ] 62  (+c1)
```

### Bar Update Logic

Each turn, the Mood bar is updated in two stages:

**Stage 1 — Player Reaction (immediate):**
The player's `a/r` + intensity rating is applied as soon as they submit it, before response selection. The bar visually updates so the player can see the effect of their internal state.

**Stage 2 — LLM Character Response:**
After the player selects a response, the LLM determines:
- Mood direction (a/r) and intensity (1/2/3) based on the chosen response
- Security direction (c/p) and intensity (1/2/3) based on the chosen response

The Stage 2 delta is applied on top of Stage 1. Both deltas are shown in the turn summary:

```
MOOD  [===============>     ] 74  (you: +a2 / them: +a1)
```

A brief inline narrative cue from the character (e.g., a shift in body language noted in italics) accompanies the update.
