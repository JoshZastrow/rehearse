## Problem Statement

realtalk is an AI conversational game framework where the game IS the data collection mechanism. Players produce natural conversational behavior through gameplay (mood arcs, turn management, win conditions), and that behavior becomes LLM training signal (SFT, DPO, trajectory examples). Layer 0 (session.py) is done — 71 tests, event-sourced session model with full training export support. v0.2 needs to add: (1) physical persistence so sessions survive crashes and support auto-resume, and (2) typed, immutable configuration so game parameters are tunable without code changes.

## Demand Evidence

The builder is the user. No external demand yet. The thesis: a conversational game that doubles as a training data flywheel. If the game is good enough to play, it's good enough to produce high-quality training data. The strongest signal is that the builder is actually building, not just planning. Four commits, working tests, real code.

## Status Quo

No current workflow for capturing conversational AI sessions as training data. The infrastructure is being built ahead of the pain. This is a yellow flag for startup viability but standard for weekend-project-turned-startup. The risk is that persistence choices won't match the actual workflow once gameplay exists.

## Target User & Narrowest Wedge

The builder themselves. The narrowest wedge is: play one full game session, have it auto-saved to disk, and export the session as SFT training examples. If that loop works end-to-end for one person, it proves the flywheel.

## Constraints

- Python 3.11+, event-sourced architecture, append-only JSONL on disk.
- Local-first: sessions live on the player's machine (~/.realtalk/sessions/). No server, no multi-user, no cloud sync.
- Workspace fingerprint isolation: sha256(cwd) = directory-scoped auto-resume.
- Config uses chz (OpenAI's immutable config library) + pydantic (JSON validation).
- Config is independent: zero imports from session.py or storage.py.
- No filesystem mocking in tests. Real disk I/O.

## Premises

1. The game IS the data flywheel: session persistence must serve BOTH real-time gameplay and batch LLM training export equally well. Neither is secondary.
2. Single-player, local-first: no server, no multi-user, no cloud sync in v0.2.
3. Workspace fingerprint model (sha256 of cwd) is the right isolation boundary.
4. chz + pydantic is the right config stack: chz for runtime objects + CLI, pydantic for deserialization/validation.
5. Config doesn't need to know about sessions: independent layers.

## Approaches Considered

### Approach A: Spec As-Written (Minimal Viable)
Build storage.py and config.py exactly as v0.2 spec describes. SessionStore + StoredSession for persistence, ConfigLoader with three-tier merge, chz classes.
- Effort: S (human: ~2 days / CC: ~30 min)
- Risk: Low
- Pros: Ships fastest. Tests pre-written.
- Cons: Rotation policy (256KB / 3 files) deletes old events. Training data loss.

### Approach B: Spec + Training-Aware Archive (Ideal Architecture)
Build the spec as-written PLUS add archive-before-rotate: before rotation deletes the oldest .jsonl.3 file, copy it to ~/.realtalk/archive/{fingerprint}/. Training exports read from archive + active. No data loss.
- Effort: S-M (human: ~3 days / CC: ~45 min)
- Risk: Low-Med
- Pros: Zero training data loss. Gameplay rotation + training archive coexist.
- Cons: Archive grows unbounded (needs policy eventually).

### Approach C: SQLite Session Store (Lateral)
Replace JSONL with SQLite database per workspace.
- Effort: M (human: ~4 days / CC: ~1 hour)
- Risk: Med
- Pros: No rotation complexity. Fast queries. ACID.
- Cons: Loses JSONL simplicity. Harder to inspect. Overkill for local game.

## Recommended Approach

**Approach B: Spec + Training-Aware Archive.** The archive addition is ~30 lines and closes the biggest gap in the spec: rotation that deletes old events means you lose training data. Since the game IS the data flywheel, losing data breaks the whole premise.

Key additions beyond the spec:
1. `archive_before_rotate()` function: copies .jsonl.3 to ~/.realtalk/archive/{fingerprint}/{timestamp}.jsonl before deletion.
2. `list_archived_sessions()` on SessionStore: returns archived files for training export.
3. Training export functions in session.py can be pointed at archive files for batch processing.

The archive directory structure:
```
~/.realtalk/
  sessions/
    a3f2c8b1e0d94f72/     <- active gameplay (rotated)
      session.jsonl
      session.jsonl.1
      session.jsonl.2
  archive/
    a3f2c8b1e0d94f72/     <- training data (never deleted)
      20260412-143000.jsonl
      20260412-150000.jsonl
```

## Design Decisions (resolved from review)

**Rotation threshold in tests:** `StoredSession` accepts an optional `max_bytes` constructor argument (default 256KB). Tests construct `StoredSession(path, max_bytes=100)` to trigger rotation without writing 256KB. The `append()` method has no `_max_bytes` parameter.

**Deep merge: lists replace, dicts merge.** When two config tiers set the same list key (e.g., both set `hooks.pre_tool_use`), the higher-priority tier's list replaces the lower tier's entirely. Dicts are merged recursively. This matches the behavior of most config systems (git, npm) and avoids ambiguity about list ordering. Add a test: two tiers both set `pre_tool_use`, higher tier wins completely.

**`PermissionMode` enum:** Defined in config.py but not referenced by any config field in v0.2. Remove it from v0.2. It belongs in v0.3+ when tool-use permissions are wired.

**`ContributorConfig.session_dir` vs `SessionStore.root`:** `ContributorConfig` owns the configured path. `SessionStore` owns disk operations. Wiring: when `contributor.enabled` is True, the game loop constructs `SessionStore(root=config.contributor.resolved_session_dir)`. They are not redundant, they have different responsibilities.

**chz version constraint:** Pin `chz>=0.3,<1.0` in pyproject.toml. The APIs used (`@chz.chz`, `chz.field`, `chz.init_property`, `chz.replace`, `chz.nested_entrypoint`) are stable across 0.3.x. If chz 1.0 breaks the API, the upper bound protects.

**Archive API contract:** Training exports should call `list_archived_sessions()` in addition to (not instead of) `list_sessions()`. The active and archive directories are disjoint. A helper `all_session_files(fingerprint)` can combine both for convenience.

## Open Questions

1. **Archive growth policy:** When does archive get cleaned up? Per-session? Per-size? Never (leave to user)? Defer to v0.3.
2. **Archive format:** Raw JSONL (same as active). Preserves optionality for different training formats.

## Success Criteria

- `pytest tests/test_storage.py tests/test_config.py` passes against real disk I/O.
- A full game session can be appended event-by-event, rotated, and reloaded across rotated files.
- Archived sessions can be read back for training export with zero data loss.
- Config loads from all three tiers with correct merge precedence.
- `chz.replace()` works for test config overrides.
- Corrupted (truncated) JSONL lines are silently skipped on load.

### Archive-Specific Acceptance Tests (Approach B)

```python
def test_archive_before_rotate_copies_oldest_file(tmp_path):
    """When rotation would delete .jsonl.3, it copies to archive first."""
    store = SessionStore(root=tmp_path)
    cwd = tmp_path / "project"
    cwd.mkdir()
    path = store.session_path(cwd)
    stored = StoredSession(path, max_bytes=100)  # low threshold
    session = new_session("claude-haiku-4-5-20251001", str(cwd))
    # Write enough events to trigger multiple rotations
    for _ in range(200):
        stored.append(session.events[0])
    archive_dir = tmp_path.parent / "archive" / path.parent.name
    # At least one archived file should exist
    assert any(archive_dir.iterdir()) if archive_dir.exists() else False

def test_list_archived_sessions_returns_paths(tmp_path):
    """list_archived_sessions returns paths to archived JSONL files."""
    store = SessionStore(root=tmp_path)
    archive_dir = tmp_path.parent / "archive" / "fakefingerprint"
    archive_dir.mkdir(parents=True)
    (archive_dir / "20260412-143000.jsonl").write_text("{}")
    paths = store.list_archived_sessions()
    assert len(paths) >= 1

def test_load_across_all_four_rotation_files(tmp_path):
    """Verify read order: .3, .2, .1, session.jsonl (oldest to newest)."""
    path = tmp_path / "session.jsonl"
    # Stage four files manually with distinct events
    # ... (each file has events with incrementing timestamps)
    stored = StoredSession(path)
    reloaded = stored.load()
    timestamps = [e.timestamp for e in reloaded.events]
    assert timestamps == sorted(timestamps)  # chronological order

def test_deep_merge_lists_replace_not_append(tmp_path, monkeypatch):
    """Two tiers both setting pre_tool_use: higher tier wins completely."""
    monkeypatch.setenv("ANTHROPIC_API_KEY", "sk-test")
    (tmp_path / ".realtalk.json").write_text(
        '{"hooks": {"pre_tool_use": ["a.sh", "b.sh"]}}'
    )
    (tmp_path / ".realtalk").mkdir()
    (tmp_path / ".realtalk" / "settings.local.json").write_text(

    config = ConfigLoader(cwd=tmp_path).load()
    assert config.hooks.pre_tool_use == ["c.sh"]  # replaced, not appended
```

## Distribution Plan

This is a Python library/CLI. Distribution via PyPI (`pip install realtalk`). No separate binary or container. Existing CI (GitHub Actions) runs tests on PRs. PyPI publishing can be added in a later spec when the package is ready for external use.

## Dependencies

- Layer 0 (session.py) must remain stable. storage.py imports from it.
- chz >= 0.3 must be available on PyPI and stable enough.
- pydantic >= 2.0 (already in pyproject.toml).

Play the game. Not "build the persistence layer" — build it AND play one full session through it. Write events by hand if the game loop doesn't exist yet. The persistence layer should feel invisible. If you notice it, it's wrong. And when the session is done, export the training examples and eyeball them. Are they what you'd actually want to train on? That's the real test.
