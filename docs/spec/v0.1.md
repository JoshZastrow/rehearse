# Realtalk — Python Build Spec v0.1

> Derived from a system design study of [ultraworkers/claw-code](https://github.com/ultraworkers/claw-code),
> a Rust implementation of a coding agent CLI.
> This spec translates those primitives into a Python project built inside-out, bottom-up.

---

## Guiding Principle

Build in dependency order. Each layer depends only on the layer below it.
Never build a consumer before you build what it consumes.
The test for "done with a layer": you can `pytest` it against real implementations
of the layer below — no mocking of filesystem, no mocking of subprocess.
The only seam you mock is the LLM API client (`ApiClient`).

## Layer 0+1 — `session.py`: Data Types + Persistence

**Build this first.** Zero dependencies. Everything else reads from or writes to a Session.

### What it owns

- `MessageRole` — enum: system / user / assistant / tool
- `ContentBlock` — tagged union: `TextBlock`, `ToolUseBlock`, `ToolResultBlock`
- `TokenUsage` — four counters: input, output, cache_creation, cache_read
- `ConversationMessage` — role + blocks + optional usage
- `Session` — the full conversation: ID, messages, model, workspace root, persistence path
- `SessionStore` — workspace-fingerprinted directory manager

### Key design decisions

**JSONL over SQLite.** Each message is one line. Append-only writes are crash-safe — a crash during write corrupts only the last incomplete line; all prior lines are intact. SQLite is worth it later if you need cross-session queries. Start here.

**Log rotation at 256KB, max 3 rotated files.** Prevents unbounded disk growth on long sessions. Pattern: `session.jsonl` → `session.jsonl.1` → `session.jsonl.2` → `session.jsonl.3` (`.4` is deleted). Active file is always `session.jsonl`.

**Workspace fingerprint for isolation.** `sha256(realpath(cwd))[:16]` becomes the session directory name. Two agents in different worktrees never collide. `--resume` is automatic — no session ID required.

**`Session.append()` writes immediately.** Don't buffer and flush. Write each message as it's appended so a crash mid-turn leaves a valid partial session.

**`Session.save()` is a full rewrite.** Only used after compaction, when the message list has been replaced. Append during normal turns; full-rewrite only on structural changes.

### Acceptance criteria

```python
# test_session.py
def test_append_survives_crash(tmp_path):
    session = Session.new_at(tmp_path / "s.jsonl", "claude-opus-4-6", tmp_path)
    session.append(ConversationMessage(MessageRole.USER, [TextBlock("hello")]))
    # Simulate crash: reload from disk
    loaded = Session.load(tmp_path / "s.jsonl")
    assert len(loaded.messages) == 1
    assert loaded.messages[0].blocks[0].text == "hello"

def test_rotation(tmp_path):
    session = Session.new_at(tmp_path / "s.jsonl", "claude-opus-4-6", tmp_path)
    # Write enough to trigger rotation
    big_text = "x" * 300_000
    session.append(ConversationMessage(MessageRole.USER, [TextBlock(big_text)]))
    assert (tmp_path / "s.jsonl.1").exists()

def test_workspace_isolation(tmp_path):
    store_a = SessionStore(tmp_path / "project_a")
    store_b = SessionStore(tmp_path / "project_b")
    assert store_a.root != store_b.root
```

## Layer 2 — `config.py`: Layered Configuration

**Build second.** No dependencies on anything above Layer 1. Loaded at startup; read throughout.

### What it owns

- `PermissionMode` — enum: read-only / workspace-write / danger-full-access / prompt
- `HookConfig` — shell command lists for PreToolUse / PostToolUse / PostToolUseFailure
- `SandboxConfig` — filesystem isolation mode + network isolation flag
- `RuntimeConfig` — the fully merged, typed config object
- `ConfigLoader` — three-tier merge: user `~/.claude.json` < project `.claude.json` < local `.claude/settings.local.json`

### Key design decisions

**Lower tiers win on conflicts.** User config is global defaults. Project config is team settings (committed to the repo). Local config is machine-specific overrides (gitignored). Exactly the git config model: `system < global < local`.

**Deep merge, not shallow.** `{"hooks": {"PreToolUse": ["a"]}}` merged with `{"hooks": {"PostToolUse": ["b"]}}` should produce `{"hooks": {"PreToolUse": ["a"], "PostToolUse": ["b"]}}` — not overwrite the hooks key entirely.

**Parse once into typed objects.** `ConfigLoader.load()` returns `RuntimeConfig`, not a raw dict. The rest of the system reads typed fields (`config.permission_mode`, `config.hooks.pre_tool_use`), not string keys. Parse errors surface at startup, not buried in a turn.

**Model resolution order:** local config `model` field beats CLI `--model` flag. Aliases in config resolve before built-in aliases (`"opus"` → `"claude-opus-4-6"`).

### Acceptance criteria

```python
def test_three_tier_merge(tmp_path):
    (tmp_path / ".claude.json").write_text('{"model": "sonnet"}')
    (tmp_path / ".claude").mkdir()
    (tmp_path / ".claude" / "settings.local.json").write_text('{"permissionMode": "read-only"}')
    config = ConfigLoader(tmp_path).load()
    assert config.model == "sonnet"
    assert config.permission_mode == PermissionMode.READ_ONLY

def test_deep_merge_hooks(tmp_path):
    (Path.home() / ".claude.json").write_text('{"hooks": {"PreToolUse": ["notify.sh"]}}')
    (tmp_path / ".claude.json").write_text('{"hooks": {"PostToolUse": ["log.sh"]}}')
    config = ConfigLoader(tmp_path).load()
    assert "notify.sh" in config.hooks.pre_tool_use
    assert "log.sh" in config.hooks.post_tool_use
```

## Layer 3 — `api.py`: LLM API Client

**Build third.** Depends on: nothing in this project. Pure I/O adapter.

### What it owns

- `AssistantEvent` — union type: `TextDelta` | `ToolUse` | `UsageEvent` | `MessageStop`
- `ApiClient` — Protocol (structural interface): `stream(request) -> Iterator[AssistantEvent]`
- `AnthropicClient` — real implementation using `anthropic` SDK
- `MockClient` — test implementation returning scripted event sequences
- `ApiRequest` — system prompt + messages + tool definitions

### Key design decisions

**`Protocol`, not `ABC`.** Any class with a `stream()` method satisfies `ApiClient` without inheriting from it. This means `MockClient` in tests doesn't need to know about the production code structure — it just needs the right method signature.

**Streaming, not batch.** The `stream()` method yields events as they arrive. `TextDelta` events reach the renderer in real time, giving the user the live typewriter effect. Batch would require waiting for the full response before rendering anything.

**Sync iterator, not async.** A `for event in client.stream(request):` loop is simpler to test and debug than `async for event in client.stream(request):`. Add `async` when multi-agent requires concurrent turns. Don't add it speculatively.

**`ToolUse.input` is a raw JSON string**, not a parsed dict. The conversation loop doesn't care about the tool's input schema — it passes the string through. The tool executor parses it. This keeps the boundary clean.

### Acceptance criteria

```python
def test_mock_client_drives_conversation():
    events = [
        TextDelta("Hello "),
        TextDelta("world"),
        UsageEvent(input_tokens=10, output_tokens=5),
        MessageStop(),
    ]
    client = MockClient(events)
    collected = list(client.stream(ApiRequest(system_prompt=[], messages=[], tools=[])))
    assert collected == events

def test_tool_use_emitted():
    events = [
        ToolUse(id="tu_1", name="Bash", input='{"command":"ls"}'),
        MessageStop(),
    ]
    client = MockClient(events)
    result = list(client.stream(...))
    assert any(isinstance(e, ToolUse) for e in result)
```

## Layer 4 — `conversation.py`: The Conversation Loop

**The most important file.** Everything else serves this.

### What it owns

- `ConversationRuntime` — the engine: `run_turn(user_input) -> TurnSummary`
- `ToolExecutor` — Protocol: `execute(tool_name, input) -> str`
- `PermissionPrompter` — Protocol: `decide(tool_name, input) -> bool`
- `TurnSummary` — iterations, usage, tool calls made, compaction info
- Auto-compaction trigger: check token count after every turn

### The turn loop

```
1. Append UserMessage to Session
2. Assemble ApiRequest (system_prompt + full session history + tool definitions)
3. stream() → collect TextDelta (render), ToolUse (queue), UsageEvent (track), MessageStop (stop)
4. Append AssistantMessage (text + tool use blocks) to Session
5. If tool uses:
     for each ToolUse:
       execute via ToolExecutor
       append ToolResult to Session
     goto 2  ← give model the results
6. If no tool uses: turn is done
7. Check token count → maybe compact
8. Return TurnSummary
```

The loop is **re-entrant by design**. The model drives itself: it calls tools until it decides it's done, then emits `MessageStop` with no tool calls. An agent implementing a complex task may loop 20–30 times before stopping.

### Key design decisions

**`ConversationRuntime` takes both `ApiClient` and `ToolExecutor` as constructor arguments.** Neither is hardcoded. In tests, inject a `MockClient` and a `FakeExecutor`. In production, inject `AnthropicClient` and `ToolRegistry`. The runtime is unaware of either implementation.

**Rebuild the runtime each turn in the CLI layer.** Between turns, the session is the only persistent state. The runtime is ephemeral. This ensures config changes, new MCP servers, or CLAUDE.md edits take effect on the next turn without restarting.

**Auto-compaction fires after the turn, not during.** The token count is checked after `MessageStop`. If over the threshold (default 100,000 input tokens, overridable via env var), `compact_session()` is called before returning `TurnSummary`. The user sees a notice; the turn result is unaffected.

**`on_text` callback for rendering.** Pass `print` in production; pass a capturing function in tests. The runtime never writes to stdout directly.

### Acceptance criteria

```python
def test_single_turn_no_tools(tmp_path):
    events = [TextDelta("Hello"), MessageStop()]
    runtime = ConversationRuntime(
        api_client=MockClient(events),
        tool_executor=NoOpExecutor(),
        session=Session.new_at(tmp_path / "s.jsonl", "test", tmp_path),
        system_prompt=["You are helpful."],
        tool_definitions=[],
        on_text=lambda *a, **kw: None,
    )
    summary = runtime.run_turn("hi")
    assert summary.iterations == 1
    assert len(runtime.session.messages) == 2  # user + assistant

def test_tool_loop(tmp_path):
    # First call: model calls Bash. Second call: model stops.
    events_turn1 = [ToolUse("t1", "Bash", '{"command":"ls"}'), MessageStop()]
    events_turn2 = [TextDelta("Done"), MessageStop()]
    runtime = ConversationRuntime(
        api_client=MockClient(events_turn1 + events_turn2),
        tool_executor=EchoExecutor(),  # returns input as output
        ...
    )
    summary = runtime.run_turn("list files")
    assert summary.iterations == 2
    assert "Bash" in summary.tool_calls
```

## Layer 5 — Tools: `permissions.py`, `hooks.py`, `bash.py`, `file_ops.py`, `tools.py`

Build these in order. Each is independent of the others except `tools.py`, which wires them together.

### `permissions.py`

**What it owns:** `PermissionPolicy` — evaluates whether a tool call is allowed given the active `PermissionMode`. Bash intent classification (read-only / write / destructive) via command prefix matching.

**Key decisions:**
- Permission is a scale, not a boolean. `ReadOnly < WorkspaceWrite < DangerFullAccess`.
- Bash gets special treatment: `rm` in `ReadOnly` mode is blocked at the intent level before any policy check runs.
- `PermissionPolicy` is a pure data object — no side effects. Pass it to `ToolRegistry`; don't let it call hooks or prompt the user directly.

### `hooks.py`

**What it owns:** `HookRunner` — fires configured shell commands at `PreToolUse`, `PostToolUse`, `PostToolUseFailure`. Returns `HookResult(decision, reason)` for pre-hooks. Post-hooks are fire-and-forget.

**Key decisions:**
- Hooks are shell commands, not Python callables. Any language, any script. This is the extension point for external systems (notification routing, CI triggers, cost dashboards).
- Exit code protocol: `0` = allow, `1` = ask, `2` = deny. Stdout is the reason string.
- `PostToolUse` uses `subprocess.Popen` (non-blocking). The turn does not wait for post-hooks.
- Hook timeout: 30 seconds. Timeout → allow. A slow hook should never block the agent.

### `bash.py`

**What it owns:** `execute_bash(input: dict, workspace_root: Path) -> str` — runs a shell command with timeout, captures stdout/stderr, enforces workspace boundary.

**Key decisions:**
- Default timeout: 120 seconds (configurable per-call via `input["timeout"]`).
- Stdout + stderr returned together, truncated at 100KB. Models have context limits; returning a 50MB build log is counterproductive.
- Background mode (`run_in_background: true`): `Popen`, return a task ID immediately, allow polling.
- Workspace boundary: prepend `cd {workspace_root} &&` unless the command is absolute-path-safe.

### `file_ops.py`

**What it owns:** `read_file`, `write_file`, `glob_search`, `grep_search` — all with workspace boundary validation.

**Key decisions:**
- 10MB read/write limit. A 10MB file would consume most of a context window anyway.
- Binary detection: read the first 8KB; if NUL bytes present, refuse with a clear error.
- `validate_workspace_boundary(path, workspace_root)`: `Path(path).resolve()` must start with `workspace_root.resolve()`. Blocks `../` traversal and symlinks that point outside the workspace.
- `read_file` returns `{content, num_lines, start_line, total_lines}` — structured, not raw string. Supports pagination for large files.

### `tools.py`

**What it owns:** `ToolRegistry` — the dispatcher. Wires `PermissionPolicy`, `HookRunner`, and individual tool handlers into a single `execute(tool_name, input) -> str` that implements `ToolExecutor`. Also owns `TOOL_DEFINITIONS` — the JSON schemas passed to the model.

**Key decisions:**
- Dispatch is a plain dict `{name: handler}`. No dynamic registration, no plugin system yet. Add entries explicitly.
- Permission check → pre-hook → execute → post-hook. Always in this order. Never skip a stage.
- Unknown tool name returns `"ERROR: unknown tool {name}"` — not an exception. The model receives this as a tool result and can react.
- MCP tools are injected into the registry at startup, same as built-in tools. The model can't tell the difference.

## Layer 6 — `prompt.py` + `compact.py`: Context and Memory

### `prompt.py`

**What it owns:** `build_system_prompt(cwd, date) -> list[str]` — assembles the final system prompt from static instructions, CLAUDE.md files, and live git context.

**Key decisions:**
- Returns a list of sections, not a single string. Sections before `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` are stable across turns and can be prompt-cached at the API level. Sections after are dynamic per-session.
- CLAUDE.md files: capped at 4,000 chars each, 12,000 chars total across all files. Protects against instruction files that balloon into full project documentation.
- Git context injected dynamically: current branch, last 5 commits, staged files. Gives the model situational awareness without requiring the user to repeat it every turn.
- Instruction precedence: `~/.claude/CLAUDE.md` (global defaults) then `./CLAUDE.md` (project-specific). Project overrides global.

### `compact.py`

**What it owns:** `compact_session(session, api_client) -> CompactionResult` — summarizes old messages when the context grows too large. Called automatically when input tokens exceed the threshold.

**Key decisions:**
- Default threshold: 100,000 input tokens. Override via `CLAW_AUTO_COMPACT_INPUT_TOKENS` env var.
- Preserve the last 4 messages verbatim. The model needs recent context to continue coherently.
- Summary is compressed to 1,200 chars / 24 lines max (`compress_summary`). Duplicate lines are removed. This prevents compaction summaries from themselves being expensive.
- Continuation preamble: `"This session is continued from a previous conversation. Continue without asking the user any further questions."` No recap, no acknowledgment.
- Compaction is transparent to the user. A notice prints; the turn result is unaffected.

## Layer 7 — `mcp.py`: External Tool Servers

Build after the core tool system is working. MCP adds capability, not architecture.

### What it owns

- `McpClient` — connects to an external MCP server process via JSON-RPC over stdio
- `McpServerManager` — manages a pool of MCP server processes; discovers their tools
- Tool name namespacing: `mcp__{server}__{tool}` (e.g. `mcp__brave__search`)
- Transport types: stdio (subprocess), SSE (HTTP stream), HTTP

### Key decisions

**stdio-first.** Most MCP servers are local subprocesses. Start with stdio transport; add SSE/HTTP later.

**Tool discovery at startup.** On `McpServerManager.start()`, send `initialize` → `tools/list` to each configured server. Register discovered tools in `ToolRegistry`. The model sees them as first-class tools.

**Namespacing prevents collisions.** Two servers can both expose a `search` tool. `mcp__brave__search` vs `mcp__filesystem__search` are unambiguous.

**Timeout: 60 seconds per tool call.** MCP servers are external processes; they can be slow or hang. Always timeout.

**Graceful degradation.** If an MCP server fails to start or hangs during discovery, log the failure and continue without it. Don't block startup.

## Layer 8 — `cli.py`: Entry Point + REPL

**Build last among core layers.** Wires everything together. Thin by design.

### What it owns

- `main()` — argparse entry point
- `run_repl()` — interactive REPL loop
- `run_prompt()` — one-shot prompt execution
- `_build_runtime()` — assembles `ConversationRuntime` from config + session + tools
- Slash command dispatch: `/exit`, `/status`, `/compact`, `/clear`, `/help`
- Output modes: text (streaming), json (batch), compact (final text only)

### Key decisions

**Rebuild `ConversationRuntime` each turn, not each session.** Between turns, only the `Session` (and its path) is persistent state. The runtime is rebuilt from the current session at the start of each turn. This ensures config changes, new MCP servers, or CLAUDE.md edits take effect immediately.

**Broad CWD safety check.** If `cwd` is `~` or `/`, warn in interactive mode and hard-error in non-interactive mode. The agent can read everything under the working directory — running from home is rarely intentional.

**Stale base preflight.** If `.realtalk-base` exists (or `--base-commit` is passed), verify that HEAD matches before starting the turn. Print a warning if not; don't block.

**Output format as a first-class concern:**
- `text` — stream `TextDelta` events to stdout as they arrive
- `json` — collect the full turn, print one JSON object at the end
- `compact` — print only the final assistant message; suppress tool output

**`/status` is local, not a turn.** It reads `Session.messages` and `UsageTracker` and prints directly. Never calls the API.

## Layer 9 — `coordination/`: Multi-Agent

Build only when the single-agent system is stable and you have a concrete use case for parallelism.

### `worker.py` — Worker Lifecycle State Machine

```
Spawning → TrustRequired → ReadyForPrompt → Running → Finished
                                                      → Failed → (1 auto-recovery) → Running
                                                               → Escalate
```

`TrustRequired` is a mandatory gate before `ReadyForPrompt`. A worker cannot receive a prompt until it has confirmed its trust context. This prevents prompt injection via the coordination channel and prompt misdelivery to an agent that hasn't finished booting.

### `tasks.py` — Task Registry + Task Packet

`TaskPacket` is the structured work contract:
```python
@dataclass
class TaskPacket:
    objective: str
    scope: str
    repo: str
    branch_policy: str
    acceptance_tests: list[str]
    commit_policy: str
    reporting_contract: str
    escalation_policy: str
```

`TaskRegistry` holds in-memory task state (Created → Running → Completed/Failed/Stopped). Not persisted — rebuilt from session history if the coordinator restarts.

### `policy.py` — Declarative Policy Engine

`PolicyRule(condition, action, priority)` — evaluated against `LaneContext`. Conditions compose (`And`, `Or`). Actions include `MergeToDev`, `Escalate`, `RecoverOnce`, `Notify`.

### `lane_events.py` — Event Vocabulary

Typed events emitted by workers, consumed by the coordinator:
`lane.started` / `lane.ready` / `lane.green` / `lane.red` / `lane.commit.created` / `lane.pr.opened` / `lane.finished` / `lane.failed` / `branch.stale_against_main`
