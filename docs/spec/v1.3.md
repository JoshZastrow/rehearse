# Realtalk — Build Spec v1.3: LLM Provider Integration via litellm.ai

> Builds on top of [v1.2](./v1.2.md).
> v1.1 established the storage and configuration layers.
> v1.2 built out configs
> v1.3 adds multi-provider LLM integration using litellm.ai with FastAPI as the abstraction layer.
> Incorporates findings from the [design review](./v1.2-design.md).

---

## What v1.3 Builds

**Layer 3:** LLM API client with multi-provider support via litellm.ai.

| Component | File | Responsibility |
|-----------|------|----------------|
| LiteLLM wrapper | `realtalk/api.py` | Abstract multi-provider LLM calls using litellm.ai |
| Request/Response types | `realtalk/api.py` | Unified protocol for streaming and non-streaming responses |
| Configuration | (uses existing `config.py`) | Provider selection, model name, API keys per provider |

**Why litellm.ai:**
- Vendor-agnostic interface across Claude, OpenAI, Gemini, Llama, local models, etc.
- Automatic fallback and retry logic
- Built-in token counting and cost estimation
- Streaming support for all providers
- Works out of the box with environment-based auth (ANTHROPIC_API_KEY, OPENAI_API_KEY, etc.)

## Current State (v1.2 Design Review Findings)

### What Works (from v1.1)
- `session.py` (Layer 0): Event-sourced, crash-safe, fully tested
- `storage.py` (Layer 1): Append-only, rotation, archival, workspace isolation
- `config.py` (Layer 2): Three-tier merge, typed, immutable

### What Needs Building (from v1.2 Design Review)
- **API stub** (`realtalk/api.py`): Currently has `AnthropicClient` stub only; no actual implementation
  - `ApiRequest` dataclass is well-designed; keep it
  - `AssistantEvent` union (TextDelta, ToolUse, UsageEvent, MessageStop) is well-designed; keep it
  - `ApiClient` protocol is correct; expand it for multi-provider
  - Need concrete implementation: `LiteLLMClient`
  - No async (v1.3 uses sync iterators; async is v2.0)

### Design Constraints (from v1.2)
1. **Sync only, no async** — game loop is single-threaded; blocking I/O is acceptable for v1
2. **Streaming is mandatory** — all responses must support streaming (tool use, tokens, end events)
3. **Provider abstraction must be seamless** — same `ApiRequest` → `Iterator[AssistantEvent]` interface for all providers
4. **No tool use in v1** — but the protocol must support it for v2.0 (litellm.ai doesn't natively handle tool use across all providers, so we handle that at our layer)

## Architecture: Multi-Provider LLM Integration

### High-Level Design

```
┌─────────────────────────────────────────────────────────────┐
│  Game Loop (conversation.py, game.py)                      │
│  - Calls: client.stream(ApiRequest) → Iterator[...Event]   │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  ApiClient Protocol (api.py)                               │
│  def stream(request: ApiRequest) → Iterator[AssistantEvent]│
└──────┬──────────────────────────────────────────────────────┘
       │
       ├─ LiteLLMClient (primary, sync)
       │  └─ litellm.completion(..., stream=True)
       │     → parses streaming response
       │     → yields AssistantEvent sequence
       │
       └─ MockClient (tests; unchanged from v1.1)
```

### LiteLLMClient: Core Implementation

```python
# realtalk/api.py

class LiteLLMClient:
    """Streaming LLM client using litellm.ai for multi-provider support.

    Supports any provider litellm.ai supports:
    - claude-3-5-sonnet, claude-opus-4-6 (Anthropic)
    - gpt-4, gpt-4-turbo (OpenAI)
    - gemini-2.0-flash (Google)
    - meta-llama/llama-2-7b (Llama via Replicate or Hugging Face)
    - ollama/model-name (local Ollama)
    - ...and others

    Environment variables (auto-detected by litellm):
    - ANTHROPIC_API_KEY       → Anthropic
    - OPENAI_API_KEY          → OpenAI
    - GOOGLE_APPLICATION_CREDENTIALS → Google
    - REPLICATE_API_TOKEN     → Replicate
    - OLLAMA_API_BASE         → Ollama

    Args:
        model: Model identifier (e.g., "claude-3-5-sonnet-20241022")
        temperature: 0.0-2.0 sampling temp
        max_tokens: Max output tokens
        api_key: Optional explicit API key (overrides env)
    """

    def __init__(
        self,
        model: str,
        temperature: float = 1.0,
        max_tokens: int = 8096,
        api_key: str | None = None,
    ) -> None:
        self.model = model
        self.temperature = temperature
        self.max_tokens = max_tokens
        self.api_key = api_key

    def stream(self, request: ApiRequest) -> Iterator[AssistantEvent]:
        """Stream events from litellm, converting to our AssistantEvent format.

        Implementation strategy:
        1. Call litellm.completion(..., stream=True)
        2. Parse streaming SSE events from response
        3. Yield TextDelta, ToolUse, UsageEvent, MessageStop as events arrive
        4. Handle provider-specific differences (streaming format, token names)
        """
        # Implementation sketch (pseudocode)
        # Actual code must handle:
        # - Anthropic: "content_block_delta" events with "type": "text_delta"
        # - OpenAI: "content_block_delta" events (different format)
        # - Streaming JSON parser for tool input accumulation
        # - Graceful error handling (malformed chunks, network errors)
        ...
```

### Request/Response Types (Keep from v1.1, Enhance)

```python
@dataclass(frozen=True)
class ApiRequest:
    """Everything the API needs to produce a response.

    platform-agnostic: works with any provider via litellm.ai.
    """
    system_prompt: list[str]        # Will be joined; allows modular system prompts
    messages: list[dict[str, object]]  # Standard OpenAI message format
    tools: list[dict[str, object]]     # Standard OpenAI tool definition format (or empty)
    model: str                          # Provider model ID (e.g., "gpt-4", "claude-opus")
    max_tokens: int = 8096
    temperature: float = 1.0

@dataclass(frozen=True)
class TextDelta:
    text: str

@dataclass(frozen=True)
class ToolUse:
    id: str
    name: str
    input: str  # raw JSON string; parser owns validation

@dataclass(frozen=True)
class UsageEvent:
    input_tokens: int
    output_tokens: int
    cache_creation_tokens: int = 0   # Anthropic prompt caching
    cache_read_tokens: int = 0       # Anthropic prompt caching

@dataclass(frozen=True)
class MessageStop:
    stop_reason: str = "end_turn"

AssistantEvent = TextDelta | ToolUse | UsageEvent | MessageStop
```

### Configuration Extension

The game loop will pass provider/model selection via `config.game.model`. Example:

```json
{
  "game": {
    "model": "claude-3-5-sonnet-20241022",
    "temperature": 1.0,
    "max_tokens": 8096
  }
}
```

The game loop (Layer 4) will instantiate:
```python
client = LiteLLMClient(
    model=config.game.model,
    temperature=config.game.temperature,
    max_tokens=config.game.max_tokens,
    api_key=config.api_key  # or None; litellm auto-detects
)
```

## Appendix: litellm.ai Primer

### Why litellm.ai?

1. **Vendor abstraction:** Same code works with Anthropic, OpenAI, Google, Llama, etc.
2. **Streaming:** All providers support streaming via litellm's unified interface
3. **Fallback & retry:** Built-in retry logic with exponential backoff
4. **Token counting:** `litellm.get_token_count()` estimates tokens for any model
5. **Cost tracking:** `litellm.completion(..., litellm_logging_obj=...)` logs costs
6. **Zero-config:** Auto-detects API keys from environment variables

### Example: Single API Call

```python
import litellm

# Automatic provider detection based on model name
response = litellm.completion(
    model="claude-3-5-sonnet-20241022",  # Anthropic is inferred
    messages=[{"role": "user", "content": "Hello"}],
    stream=True,  # Enable streaming
)

for event in response:
    if event.get("choices"):
        delta = event["choices"][0].get("delta", {})
        if "content" in delta:
            print(delta["content"], end="", flush=True)
```

### litellm Event Format

When `stream=True`, litellm yields SSE events as dicts:

```json
// Anthropic-style (litellm normalizes this)
{"type": "content_block_delta", "delta": {"type": "text_delta", "text": "Hello"}}
{"type": "message_stop", "message": {"content": [...]}}

// OpenAI-style (also normalized by litellm)
{"choices": [{"delta": {"role": "assistant", "content": "Hello"}}]}
{"choices": [{"finish_reason": "stop"}]}
```

Our `LiteLLMClient` normalizes all of this into our internal `AssistantEvent` union.

## Appendix: Testing with MockClient

The game loop and conversation engine can be fully tested without hitting any real API:

```python
from realtalk.api import MockClient, TextDelta, MessageStop, ApiRequest

# Simulate a complete response
events = [
    TextDelta("Hello, I'm "),
    TextDelta("happy to meet you."),
    MessageStop(),
]

client = MockClient(events)
request = ApiRequest(
    system_prompt=["You are kind."],
    messages=[{"role": "user", "content": "Hi"}],
    tools=[],
    model="mock"
)

# Game loop or conversation engine calls:
for event in client.stream(request):
    if isinstance(event, TextDelta):
        print(f"Text: {event.text}")
    elif isinstance(event, MessageStop):
        print("Done!")
```

This pattern ensures:
- **Fast tests** (no API latency)
- **Deterministic** (no API variability)
- **Cheap** (no API costs)
- **Works offline** (no network needed)

- ✅ 23 new tests (unit + integration)

- Async `AsyncLiteLLMClient` for concurrent requests
