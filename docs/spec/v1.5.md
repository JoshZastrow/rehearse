# Realtalk — Build Spec v1.5: Hook Runner (Layer 5)

> Builds on [v1.4](./v1.4.md) (conversation runtime).
> Layer 4 delivered `ConversationRuntime` with the `ToolExecutor` protocol and tool loop.
> **v1.5 builds the hook runner** — the shell-command extension point that fires before
> and after every tool call, allowing operators to observe, gate, or log tool execution.
>
> The hook system is adapted from the [Rust implementation in claw-code](https://github.com/ultraworkers/claw-code),
> a coding agent CLI. This spec translates those patterns into idiomatic Python.

---

## What Exists

### Layer 2: `config.py`

The hook configuration is already defined and loaded by the three-tier config system:

```python
@chz.chz
class HookConfig:
    pre_tool_use: list[str] = chz.field(default_factory=list)
    post_tool_use: list[str] = chz.field(default_factory=list)
    post_tool_use_failure: list[str] = chz.field(default_factory=list)
```

Each field is a list of shell command strings. An empty list means no hooks for that event.
Configured in `.realtalk.json` or `~/.realtalk/config.json`:

```json
{
  "hooks": {
    "pre_tool_use": ["python3 /opt/hooks/log_tool_call.py"],
    "post_tool_use": ["curl -s https://metrics.example.com/tool_complete"],
    "post_tool_use_failure": ["notify-send 'Tool failed'"]
  }
}
```

### Layer 4: `conversation.py`

The `ConversationRuntime` calls `executor.execute(tool_name, tool_input)` for each `ToolUse` event from the LLM. The hook runner will be called by the `ToolRegistry` (a future Layer 5 module) which sits between the runtime and the actual tool handlers:

```
ConversationRuntime
  └─ ToolRegistry.execute()    # implements ToolExecutor
       ├─ HookRunner.run_pre_tool_use()
       ├─ handler(tool_input)
       └─ HookRunner.run_post_tool_use()  or  run_post_tool_use_failure()
```

## Reference Design: The Rust Hook System

The Rust implementation (~988 LOC) in claw-code establishes the hook execution model. Understanding it explains why v1.5 is shaped the way it is.

### Event Lifecycle

Three hook points bracket every tool call:

```
PreToolUse  →  [tool executes]  →  PostToolUse
                                  →  PostToolUseFailure (on error)
```

`PreToolUse` is the only point that can **deny** execution — it fires before the tool runs and can veto it. `PostToolUse` and `PostToolUseFailure` are observational — they see what happened but cannot change the outcome.

The Rust code models this as an enum:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum HookEvent {
    PreToolUse,
    PostToolUse,
    PostToolUseFailure,
}
```

### Shell Command Execution

Each hook is a shell command string run via `sh -lc <command>` (or `cmd /C` on Windows). The `-l` flag loads the user's login shell profile so PATH, environment variables, and aliases are available.

The command receives context two ways:

**1. Environment variables** — lightweight, available without parsing:

```rust
child.env("HOOK_EVENT", event.as_str());
child.env("HOOK_TOOL_NAME", tool_name);
child.env("HOOK_TOOL_INPUT", tool_input);
child.env("HOOK_TOOL_IS_ERROR", if is_error { "1" } else { "0" });
if let Some(tool_output) = tool_output {
    child.env("HOOK_TOOL_OUTPUT", tool_output);
}
```

**2. Stdin JSON payload** — full structured context for hooks that need to parse:

```rust
fn hook_payload(event, tool_name, tool_input, tool_output, is_error) -> Value {
    json!({
        "hook_event_name": event.as_str(),
        "tool_name": tool_name,
        "tool_input": parse_tool_input(tool_input),  // parsed JSON
        "tool_input_json": tool_input,                 // raw string
        "tool_output": tool_output,
        "tool_result_is_error": is_error,
    })
}
```

Note the dual representation: `tool_input` is parsed JSON (for easy `jq` access in shell scripts) while `tool_input_json` is the raw string (for hooks that need exact fidelity).

### Exit Code Protocol

The hook communicates its decision via exit code:

- **`0`** — allow (continue execution)
- **`2`** — deny (block the tool call; stdout is the reason string)
- **Any other code** — failure (hook itself broke; logged, stops the chain)

Why exit code 2 for deny (not 1)? Because `1` is the default exit for uncaught errors in most languages. A hook that crashes should be treated as a failure, not as a deliberate denial. This prevents accidental blocking.

```rust
match output.status.code() {
    Some(0) => HookCommandOutcome::Allow { parsed },
    Some(2) => HookCommandOutcome::Deny { parsed },
    Some(code) => HookCommandOutcome::Failed { parsed },
    None => HookCommandOutcome::Failed { /* terminated by signal */ },
}
```

### Sequential Execution with Early Termination

Multiple hooks for the same event run in config order. On the first denial or failure, execution stops — later hooks do not run:

```rust
for command in commands {
    match run_command(command, ...) {
        Allow { parsed }  => merge_into_result(parsed),
        Deny { parsed }   => { result.denied = true; return result; }
        Failed { parsed } => { result.failed = true; return result; }
        Cancelled { msg } => { result.cancelled = true; return result; }
    }
}
```

### Result Aggregation

The Rust `HookRunResult` accumulates messages from all hooks that ran:

```rust
struct HookRunResult {
    denied: bool,
    failed: bool,
    cancelled: bool,
    messages: Vec<String>,
    permission_override: Option<PermissionOverride>,
    permission_reason: Option<String>,
    updated_input: Option<String>,
}
```

Messages are collected from stdout of each hook. The `merge_parsed_hook_output` function appends messages and promotes the last-seen permission/input overrides.

### Structured JSON Output (v2 — Not Implemented in v1.5)

The Rust implementation parses hook stdout as JSON when possible, looking for structured fields:

```rust
fn parse_hook_output(stdout: &str) -> ParsedHookOutput {
    // Attempts JSON parse; falls back to raw string
    // Looks for: systemMessage, reason, continue, decision,
    //            hookSpecificOutput.permissionDecision,
    //            hookSpecificOutput.updatedInput
}
```

This allows hooks to override permissions (`"permissionDecision": "allow"`) or rewrite tool input (`"updatedInput": {"command": "git status"}`). These are coding-agent features — a hook that rewrites a dangerous bash command into a safe one. RealTalk v1 does not need this; exit codes and stdout strings are sufficient.

### Thread-Safe Cancellation (v2 — Not Implemented in v1.5)

The Rust version supports `HookAbortSignal` — an `AtomicBool` that a separate thread can set to cancel a long-running hook:

```rust
pub struct HookAbortSignal {
    aborted: Arc<AtomicBool>,
}
```

The `run_command` function polls `try_wait()` in a loop with 20ms sleeps, checking the abort signal between polls. If aborted, the child process is killed.

Python's single-threaded game loop has no concurrent cancellation source. The `subprocess.run` timeout parameter provides the same protection against hung hooks.

### Progress Reporting (v2 — Not Implemented in v1.5)

The Rust version has a `HookProgressReporter` trait that fires `Started`, `Completed`, and `Cancelled` events for each hook command. This drives UI indicators in the coding agent. RealTalk hooks are invisible to the player — no progress UI needed.

## Python Implementation

### Type Definitions

```python
from __future__ import annotations

import json
import os
import subprocess
from dataclasses import dataclass
from enum import StrEnum

from realtalk.config import HookConfig


class HookEvent(StrEnum):
    """Points in the tool execution lifecycle where hooks fire."""
    PRE_TOOL_USE = "PreToolUse"
    POST_TOOL_USE = "PostToolUse"
    POST_TOOL_USE_FAILURE = "PostToolUseFailure"


@dataclass(frozen=True)
class HookContext:
    """Context passed to hook commands via env vars and stdin JSON.

    Always populated: tool_name, tool_input.
    Populated for post-hooks only: tool_output, tool_is_error.
    """
    tool_name: str
    tool_input: str            # raw JSON string from the LLM
    tool_output: str = ""      # tool result (post-hooks only)
    tool_is_error: bool = False


@dataclass(frozen=True)
class HookResult:
    """Result of running all hooks for a single event.

    Only PreToolUse hooks can deny. Post-hooks populate messages only.
    A failed result means the hook itself broke (non-0, non-2 exit or timeout).
    """
    denied: bool = False
    failed: bool = False
    reason: str = ""                # denial/failure reason (from stdout)
    messages: tuple[str, ...] = ()  # system messages from all hooks that ran
```

### HookRunner

```python
class HookRunner:
    """Runs configured shell commands for hook events.

    Commands execute via ``sh -lc <command>`` with environment variables
    and a JSON payload on stdin. Exit code protocol:

        0 = allow (continue)
        2 = deny (block the tool call; stdout is the reason)
        other = failure (logged, chain stops)

    Hooks run sequentially in config order. Stop on first deny or failure.

    Timeout default: 30 seconds. A slow hook never blocks the game.
    ``subprocess.TimeoutExpired`` is caught and treated as failure (not denial).

    Adapted from the Rust ``HookRunner`` in claw-code. The Rust version
    handles thread-safe cancellation (``HookAbortSignal``), progress
    reporting (``HookProgressReporter``), and structured JSON output
    parsing — all deferred to v2 in Python.
    """

    def __init__(self, config: HookConfig, timeout: float = 30.0) -> None:
        self._config = config
        self._timeout = timeout

    def run_pre_tool_use(self, context: HookContext) -> HookResult:
        """Fire PreToolUse hooks. Can deny the tool call."""
        return self._run_commands(
            HookEvent.PRE_TOOL_USE,
            self._config.pre_tool_use,
            context,
        )

    def run_post_tool_use(self, context: HookContext) -> HookResult:
        """Fire PostToolUse hooks. Observational — cannot deny."""
        return self._run_commands(
            HookEvent.POST_TOOL_USE,
            self._config.post_tool_use,
            context,
        )

    def run_post_tool_use_failure(self, context: HookContext) -> HookResult:
        """Fire PostToolUseFailure hooks. Observational."""
        return self._run_commands(
            HookEvent.POST_TOOL_USE_FAILURE,
            self._config.post_tool_use_failure,
            context,
        )

    def _run_commands(
        self,
        event: HookEvent,
        commands: list[str],
        context: HookContext,
    ) -> HookResult:
        """Execute commands sequentially. Stop on first deny or failure."""
        if not commands:
            return HookResult()

        messages: list[str] = []
        payload = self._build_payload(event, context)
        env = self._build_env(event, context)

        for command in commands:
            result = self._run_one(command, payload, env)
            if result.messages:
                messages.extend(result.messages)
            if result.denied:
                return HookResult(
                    denied=True,
                    reason=result.reason,
                    messages=tuple(messages),
                )
            if result.failed:
                return HookResult(
                    failed=True,
                    reason=result.reason,
                    messages=tuple(messages),
                )

        return HookResult(messages=tuple(messages))

    def _run_one(
        self,
        command: str,
        payload: str,
        env: dict[str, str],
    ) -> HookResult:
        """Run a single shell command and interpret the exit code."""
        try:
            proc = subprocess.run(
                ["sh", "-lc", command],
                input=payload,
                capture_output=True,
                text=True,
                timeout=self._timeout,
                env=env,
            )
        except subprocess.TimeoutExpired:
            return HookResult(
                failed=True,
                reason=f"Hook `{command}` timed out after {self._timeout}s",
                messages=(f"Hook `{command}` timed out",),
            )
        except OSError as exc:
            return HookResult(
                failed=True,
                reason=f"Hook `{command}` failed to start: {exc}",
                messages=(f"Hook `{command}` failed to start: {exc}",),
            )

        stdout = proc.stdout.strip()

        if proc.returncode == 0:
            msgs = (stdout,) if stdout else ()
            return HookResult(messages=msgs)

        if proc.returncode == 2:
            reason = stdout or f"Hook `{command}` denied the tool call"
            return HookResult(denied=True, reason=reason, messages=(reason,))

        # Any other exit code = failure
        reason = (
            stdout
            or proc.stderr.strip()
            or f"Hook `{command}` exited with status {proc.returncode}"
        )
        return HookResult(failed=True, reason=reason, messages=(reason,))

    def _build_payload(self, event: HookEvent, ctx: HookContext) -> str:
        """Build the JSON payload piped to the hook's stdin.

        Mirrors the Rust ``hook_payload()`` function. ``tool_input`` is
        parsed JSON (for ``jq`` access); ``tool_input_json`` is raw.
        """
        payload: dict[str, object] = {
            "hook_event_name": event.value,
            "tool_name": ctx.tool_name,
            "tool_input": _parse_tool_input(ctx.tool_input),
            "tool_input_json": ctx.tool_input,
        }
        if event == HookEvent.POST_TOOL_USE_FAILURE:
            payload["tool_error"] = ctx.tool_output
            payload["tool_result_is_error"] = True
        else:
            payload["tool_output"] = ctx.tool_output or None
            payload["tool_result_is_error"] = ctx.tool_is_error
        return json.dumps(payload)

    def _build_env(self, event: HookEvent, ctx: HookContext) -> dict[str, str]:
        """Build environment variables for the hook subprocess."""
        env = os.environ.copy()
        env["HOOK_EVENT"] = event.value
        env["HOOK_TOOL_NAME"] = ctx.tool_name
        env["HOOK_TOOL_INPUT"] = ctx.tool_input
        env["HOOK_TOOL_IS_ERROR"] = "1" if ctx.tool_is_error else "0"
        if ctx.tool_output:
            env["HOOK_TOOL_OUTPUT"] = ctx.tool_output
        return env


def _parse_tool_input(raw: str) -> object:
    """Parse tool input JSON for the payload. Falls back to ``{"raw": raw}``."""
    try:
        return json.loads(raw)
    except (json.JSONDecodeError, TypeError):
        return {"raw": raw}
```

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Exit code 2 = deny, not 1 | Match Rust convention | Exit 1 is the default for uncaught errors in most languages. A crashing hook should be a failure, not an accidental denial |
| Failure stops the chain | Same as deny | A broken hook indicates something unexpected. Safer to halt than continue with unknown state. Matches the Rust `Failed` arm returning early |
| Timeout = failure, not deny | `TimeoutExpired` → `failed=True` | A slow hook is broken infrastructure, not a deliberate policy decision. A timeout should never block the game |
| Post-hooks are blocking | `subprocess.run`, not `Popen` | Simplifies v1. Post-hooks are rarely configured in a game context. Non-blocking post-hooks via `Popen` are a v2 optimization |
| Stdin payload has dual input repr | `tool_input` (parsed) + `tool_input_json` (raw) | Matches the Rust payload. Parsed JSON is convenient for `jq` in shell scripts; raw string preserves fidelity |
| Skip structured JSON stdout parsing | Exit code + raw stdout only | v1 hooks communicate via exit codes. Structured output fields (`permissionDecision`, `updatedInput`, `systemMessage`) are coding-agent extensions |
| Skip `HookAbortSignal` | No thread-safe cancellation | Python game loop is single-threaded. `subprocess.run(timeout=...)` handles hung hooks |
| Skip `HookProgressReporter` | No hook progress UI | Hooks are invisible to the player. No UI to drive |
| `sh -lc` for command execution | Login shell with profile | Ensures PATH and env vars from `.bashrc`/`.zshrc` are available. Matches the Rust `shell_command()` function |

## Acceptance Criteria

Tests use **real subprocesses** (`sh -c '...'`). This matches the project's testing philosophy: the only seam you mock is the LLM API client.

```python
def test_empty_hooks_return_default():
    runner = HookRunner(HookConfig())
    ctx = HookContext(tool_name="update_mood", tool_input='{"delta": 5}')
    result = runner.run_pre_tool_use(ctx)
    assert not result.denied
    assert not result.failed
    assert result.messages == ()


def test_pre_hook_allow_exit_0():
    config = HookConfig(pre_tool_use=["printf 'hook ok'"])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert not result.denied
    assert "hook ok" in result.messages


def test_pre_hook_deny_exit_2():
    config = HookConfig(pre_tool_use=["printf 'blocked'; exit 2"])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert result.denied
    assert result.reason == "blocked"


def test_pre_hook_failure_exit_1():
    config = HookConfig(pre_tool_use=["printf 'oops'; exit 1"])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert result.failed
    assert not result.denied


def test_sequential_stop_on_deny():
    """First hook allows, second denies, third never runs."""
    config = HookConfig(pre_tool_use=[
        "printf 'first ok'",
        "printf 'denied'; exit 2",
        "printf 'never runs'",
    ])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert result.denied
    assert "first ok" in result.messages
    assert not any("never runs" in m for m in result.messages)


def test_sequential_stop_on_failure():
    """Failure also stops the chain — later hooks do not run."""
    config = HookConfig(pre_tool_use=[
        "printf 'broken'; exit 1",
        "printf 'never runs'",
    ])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert result.failed
    assert not any("never runs" in m for m in result.messages)


def test_env_vars_set():
    config = HookConfig(pre_tool_use=["echo $HOOK_TOOL_NAME"])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert "update_mood" in result.messages


def test_stdin_json_payload():
    """Hook reads stdin and verifies the JSON payload structure."""
    script = "python3 -c 'import sys,json; d=json.load(sys.stdin); print(d[\"tool_name\"])'"
    config = HookConfig(pre_tool_use=[script])
    runner = HookRunner(config)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{"delta": 5}')
    )
    assert "update_mood" in result.messages


def test_timeout_treated_as_failure():
    config = HookConfig(pre_tool_use=["sleep 5"])
    runner = HookRunner(config, timeout=0.5)
    result = runner.run_pre_tool_use(
        HookContext(tool_name="update_mood", tool_input='{}')
    )
    assert result.failed
    assert not result.denied


def test_post_hook_receives_output():
    config = HookConfig(post_tool_use=["echo $HOOK_TOOL_OUTPUT"])
    runner = HookRunner(config)
    result = runner.run_post_tool_use(
        HookContext(
            tool_name="update_mood",
            tool_input='{}',
            tool_output='{"mood": 65}',
        )
    )
    assert '{"mood": 65}' in result.messages


def test_post_failure_hook_receives_error_flag():
    config = HookConfig(post_tool_use_failure=["echo $HOOK_TOOL_IS_ERROR"])
    runner = HookRunner(config)
    result = runner.run_post_tool_use_failure(
        HookContext(
            tool_name="update_mood",
            tool_input='{}',
            tool_output="ERROR: ValueError",
            tool_is_error=True,
        )
    )
    assert "1" in result.messages
```

## What Waits for v2

| Feature | Rust Reference | Why Deferred |
|---------|---------------|--------------|
| Structured JSON stdout parsing | `parse_hook_output()` with `systemMessage`, `permissionDecision`, `updatedInput` | Coding-agent extension. Game hooks use exit codes only |
| `HookAbortSignal` | `Arc<AtomicBool>` polled in 20ms loop | Single-threaded game loop; `subprocess.run(timeout=...)` suffices |
| `HookProgressReporter` | `Started`/`Completed`/`Cancelled` trait events | No hook progress UI in the game |
| Non-blocking post-hooks | `Popen` (fire-and-forget) | Blocking is simpler; post-hooks are rare in game context |
| `permission_override` / `updated_input` on result | `HookRunResult` fields | Hooks cannot modify tool input or override permissions in v1 |
