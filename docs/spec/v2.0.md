## Requirements

### R1. A player types `realtalk` and is playing within 5 seconds.

No flags, no config file, no setup wizard. `ANTHROPIC_API_KEY` in the environment is the only prerequisite. Everything else has defaults.

### R2. The game is a 10-25 turn conversation with a reactive AI character.

The player picks a scene, picks a role, reads the character's opening line, then enters the game loop: rate reaction, pick a response, watch bars move, repeat. Win by reaching mood >= 80 and passing the Invitation Arc. Lose by tanking mood to 0 or hitting the 25-turn cap.

### R3. Every game session produces RL-compatible training data.

The game loop maps cleanly onto an MDP (Markov Decision Process). Each turn produces a `(state, action, reward, next_state)` tuple. The session JSONL already captures this via the event-sourced model. Contributor mode writes the training-specific annotations (player reaction ratings, option labels) that the base session does not include.

### R4. The TUI renders in any terminal >= 80 columns.

Full-screen layout: dialogue area, mood/security bars with color + numeric values, input area. No external runtime dependency beyond Node.js (for ink) or Python (for textual). The TUI must degrade gracefully with `--no-color`.

### R5. Compaction prevents context window overflow on long sessions.

Even at 25 turns, a session with tool calls can reach 15-20K tokens. If the player replays or the LLM is verbose, we want a safety net. Compact at a configurable threshold (default 80K input tokens), preserve the last 4 messages, and continue transparently.

## Assumptions (to be validated)

### A1. Textual (Python) is the TUI framework.

**Rationale:** The entire backend is Python. Adding Node.js/ink creates a cross-process bridge (Python game engine <-> Node.js renderer) with JSON-RPC or stdio pipes between them. That's a second runtime, a second package manager, and a new failure mode for every player.

Textual gives us: single `pip install realtalk`, single process, direct function calls between engine and UI, async rendering via asyncio, CSS-like styling, and widget library (progress bars, text areas, input fields). The tradeoff is that Textual's component model is less elegant than React/ink, but we're building 5 screens, not a design system.

**Decision: Use Textual.** Revisit ink only if Textual can't render the bars or streaming text acceptably.

### A2. The game state is a discrete MDP with well-defined S, A, R, T.

**Rationale:** MuZero operates on three learned functions:

| MuZero Function | Game Equivalent | What it does |
|---|---|---|
| **Representation h(o)** -> s | `GameState.snapshot()` -> `LatentState` | Maps raw observation (mood, security, turn, arc, history embedding) into a fixed-size state vector |
| **Dynamics g(s, a)** -> (s', r) | `step(state, action)` -> `(next_state, reward)` | Given current state + player action, predicts next state and immediate reward |
| **Prediction f(s)** -> (p, v) | `predict(state)` -> `(policy, value)` | From a state, predicts which action is best (policy) and expected total reward (value) |

For this to work, we need:
- **State (S):** A serializable snapshot of everything the game knows at turn N. Mood, security, turn number, arc status, and a fixed-length embedding of conversation history.
- **Action (A):** The player's discrete choice. This is a composite: `(reaction_direction, reaction_intensity, response_choice)` = `{a,r} x {1,2,3} x {1,2,3}` = 18 possible actions per turn.
- **Reward (R):** Immediate scalar feedback. Mood delta + security delta + win/loss bonus.
- **Transition (T):** Deterministic from the game mechanics (reaction deltas are fixed), stochastic from the LLM (character response, next options).

The LLM is the **environment**, not the agent. The player is the agent. MuZero would learn to play the game by predicting what the LLM-environment will do next.

**Decision: Structure the game loop as an MDP step function.** Each turn calls `step(state, action) -> (next_state, reward, done, info)` matching the Gymnasium API. The game engine is the environment. The TUI is the human policy interface. Contributor mode logs the trajectory.

### A3. Compaction is worth building even for short sessions.

**Rationale:** The main session is 10-25 turns. But each turn has 2-4 tool calls, each with structured JSON input/output. A 20-turn game with tool calls easily reaches 30K tokens. Add verbose character dialogue and the system prompt (~2-3K), and we're at 40K+. If a player replays or the model gets chatty, 80K is reachable. Compaction costs ~200 LOC and provides a safety net that prevents a hard crash.

**Decision: Build compaction.** Threshold at 80K input tokens. Preserve last 4 messages. Use a lightweight summarization call.

### A4. The system prompt has two halves: static (cacheable) and dynamic (per-session).

**Rationale:** Following the claw-code pattern, everything before `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` is identical across sessions and can be prompt-cached (Anthropic caches matching prefixes for 5 minutes, reducing cost by 90% on the cached portion). The character personality, scene context, and current game state go after the boundary.

For a game that costs ~$0.01-0.05 per session on Haiku, caching isn't urgent. But the architectural split costs nothing and makes prompt debugging much easier (you can diff the static half once and focus on the dynamic half).

**Decision: Two-section prompt with boundary marker.** Static = game rules, tool instructions, response format. Dynamic = scene, role, game state, turn context.

## First Principles Evaluation

### Does the MuZero framing actually help, or is it premature abstraction?

MuZero needs three things from the game: (1) a state representation, (2) a step function, and (3) logged trajectories.

(1) is just `GameState` serialized, which we have. (2) is the game loop refactored as a pure function, which is a better design regardless of RL. (3) is contributor mode, which is already built.

The cost of MuZero alignment is: defining `StateSnapshot` (a frozen dataclass), wrapping the turn logic in a `step()` function, and logging `(s, a, r, s')` tuples. Maybe 50 lines on top of what we'd write anyway. The alternative, building a game loop that doesn't separate state transitions from rendering, would require a rewrite later.

**Verdict: Do it.** The MDP step function is just clean architecture with a name. We're not building neural networks or MCTS. We're structuring the data so they're possible later.

### Is Textual actually good enough for this UI?

The UI has 5 screens:
1. Scene selection (3 options, arrow keys)
2. Role selection (5 options, arrow keys)
3. Situation + opening line (text display, press enter)
4. Game loop (bars + dialogue + reaction input + option selection)
5. Post-game summary (text display, play again y/n)

Textual handles all of these. It has `OptionList` for selection screens, `RichLog` for streaming dialogue, `ProgressBar` for mood/security, `Input` for text entry, and CSS for layout. The game loop screen is the most complex, and it maps directly to a Textual `Screen` with 3-4 widgets.

**Verdict: Textual is sufficient.** The game's UI complexity is well within what Textual handles daily.

### Should compaction call the LLM or use a heuristic?

Two options:
- **LLM summarization:** Send old messages to the model, get a summary back. Accurate but adds latency and cost.
- **Heuristic truncation:** Drop messages older than N turns, keep the system prompt and last 4 messages. Cheap but loses context.

For a 10-25 turn game, heuristic truncation is probably fine. The LLM has the full game state in the system prompt (mood, security, turn number, arc status), so losing early dialogue isn't catastrophic. The character's personality comes from the system prompt, not from remembering turn 3.

**Verdict: Heuristic truncation first.** Keep last 4 messages + system prompt. Add LLM summarization as an upgrade if players report the character "forgetting" things.

## Layer 4 Amendments -- `conversation.py`

The engine needs to swap the system prompt and session on the running
`ConversationRuntime` between turns (per-turn prompt refresh and compaction
swap). Add two setters:

```python
class ConversationRuntime:
    ...

    def set_system_prompt(self, sections: list[str]) -> None:
        """Replace the system prompt used on the next turn.

        The engine calls this before every `run_turn` so the LLM always sees
        the current GameState + compacted summary sections.

    def set_session(self, session: Session) -> None:
        """Replace the current session (used after compaction).

        self._session = session

## Layer 6a -- `prompt.py`: System Prompt Builder

### What it owns

- `SystemPromptBuilder` -- builder pattern, returns `list[str]` (sections)
- `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` -- marker string separating cacheable from dynamic
- `build_character_prompt(scene, role, game_state)` -> dynamic section
- `build_game_rules_prompt()` -> static section (tool usage rules, response format, turn structure)
- `build_state_context(game_state)` -> dynamic section (current mood, security, turn, arc status)

### Design (following claw-code prompt.rs)

The builder produces a `list[str]` where each element is one prompt section. Sections before the boundary marker are **static**: identical across all sessions with the same game version. Sections after are **dynamic**: scene, role, character personality, and live game state.

```python
@dataclass(frozen=True)
class PromptSection:
    """One section of the system prompt."""
    content: str
    cacheable: bool  # True = before boundary, False = after

class SystemPromptBuilder:
    """Assembles the system prompt from static rules + dynamic context."""

    def build(
        self,
        scene: Scene,
        role: Role,
        game_state: GameState,
    ) -> list[str]:
        sections = []
        # Static half (cacheable across sessions)
        sections.append(self._game_identity())
        sections.append(self._tool_instructions())
        sections.append(self._response_format_rules())
        sections.append(self._content_guidelines())
        sections.append(SYSTEM_PROMPT_DYNAMIC_BOUNDARY)
        # Dynamic half (per-session)
        sections.append(self._scene_context(scene))
        sections.append(self._role_personality(role))
        sections.append(self._game_state_context(game_state))
        sections.append(self._turn_instructions(game_state))
        return sections
```

### Static sections (before boundary)

**Game identity:** "You are a character in RealTalk, a conversational game. You are a living person, not an AI. Never break character. Never reference being an AI, having a system prompt, or the game mechanics."

**Tool instructions:** Explain what each tool does, when to call them, and the expected response format. "After the player selects a response, call `character_respond` with your dialogue, mood/security deltas, and whether this is an invitation turn. Then call `generate_options` with 3 new response options."

**Response format rules:** "Always respond in character. Dialogue should be 1-4 sentences. Narrative cues (body language, environment) should be in italics and kept to one sentence."

**Content guidelines:** From the PRD. No explicit content, no coercion, no stereotypes.

### Dynamic sections (after boundary)

**Scene context:** "You are in: {scene.description}. The atmosphere is {scene.atmosphere}."

**Role personality:** "Your relationship to the player: {role.name}. Key traits: {role.traits}. Win-condition tone: {role.win_tone}."

**Game state context:** "Current game state -- Mood: {mood}/100. Security: {security}/100. Turn: {turn_number}. Arc active: {arc_active}. {arc_context if arc_active}."

**Turn instructions:** Vary by game phase. Early game: "Build rapport. React naturally." Arc active: "Your dialogue should shift tonally. Reference something personal. Build toward a genuine opening." Invitation turn: "This is the moment. Deliver a genuine, role-appropriate invitation."

## Layer 6b -- `compact.py`: Session Compaction

> Reference: [claw-code compact.rs](https://github.com/ultraworkers/claw-code/blob/main/rust/crates/runtime/src/compact.rs)
> Ported to Python, adapted for our event-sourced Session model.

### What it owns

- `CompactionConfig` -- frozen config: `preserve_recent_messages`, `max_estimated_tokens`
- `CompactionResult` -- frozen result: `summary_sections`, `formatted_summary`, `compacted_session`, `removed_message_count`
- `estimate_event_tokens(event) -> int` -- per-event token estimate (len/4 + 1)
- `estimate_session_tokens(session) -> int` -- sum across all events
- `should_compact(session, config, existing_summary_sections) -> bool` -- threshold check
- `compact_session(session, config, existing_summary_sections) -> CompactionResult` -- the main function
- `format_compact_summary(summary) -> str` -- strip analysis tags, format summary tags
- `build_summary_sections(summary, recent_preserved) -> list[str]` -- wraps summary + preamble + resume note into `system_prompt` sections
- `summarize_messages(messages) -> str` -- structured summary builder
- `merge_compact_summaries(existing, new) -> str` -- handles re-compaction
- `_rebuild_session(original, preserved_messages) -> Session` -- event-preserving session rebuild

### Design (following compact.rs, adapted for system-prompt delivery)

> **Important deviation from compact.rs:** claw-code's Rust implementation
> inserts a synthetic *system-role message* at the head of the rebuilt
> conversation. We do NOT. The Anthropic API rejects `role: system` inside
> the `messages` array, and our Session event log has no `add_system_text`.
> Instead, the compacted summary is returned as `list[str]` sections that the
> engine prepends to `system_prompt` on every subsequent turn via
> `runtime.set_system_prompt(...)`.

No LLM call. Compaction builds a **structured summary** of removed messages,
then produces (1) a list of `system_prompt` sections carrying that summary
plus the continuation preamble, and (2) a rebuilt Session containing ONLY
the preserved recent messages (plus their paired tool events).

**Key behaviors ported from claw-code:**

1. **Structured summary, not just truncation.** The summary includes: message scope (counts by role), tool names mentioned, recent user requests (last 3), pending work (messages containing "todo"/"next"/"remaining"), and a key timeline (one line per removed message).

2. **Tool-use / tool-result boundary protection.** When choosing the compaction boundary, never split an assistant ToolUse from its paired user ToolResult. If the first preserved message starts with a ToolResult, walk the boundary backward until the pair is intact. Without this, the API rejects the orphaned tool result with a 400.

3. **Re-compaction merges summaries.** The engine passes the *previous* summary sections back into `compact_session` as `existing_summary_sections: list[str] | None`. If present, the merger extracts highlights and concatenates "Previously compacted context" + "Newly compacted context" sections. (The summary is NOT stored in `session.events` -- it lives on `GameEngine` and is replayed into `system_prompt` each turn.)

4. **Existing summary excluded from threshold check.** `should_compact()` does not scan the session for summary prefixes (the Session has none). The engine passes `existing_summary_sections` so the threshold math can discount already-summarized tokens if desired. For v2.0 the threshold is based purely on the compactable Session tokens.

5. **Continuation preamble with flags.** The summary sections list includes the preamble, the summary body, the recent-messages note, and the direct-resume instruction -- each as a separate list element so prompt caching can share the boundary.

         `run_turn` calls see the slimmed message history.
      3. Pass `result.summary_sections` back into the NEXT
         `compact_session(..., existing_summary_sections=...)` call so
         re-compaction can merge highlights.
    """
    summary_sections: list[str]  # prepend to system_prompt on next turn
    formatted_summary: str       # human-readable (tags stripped) -- for logs / debug
    compacted_session: Session   # rebuilt Session with only preserved messages + paired tool events
    removed_message_count: int   # how many messages were dropped
```

### Token estimation

Walks session events, not derived messages. Each event contributes based on its text content:

```python
def _estimate_event_tokens(event: SessionEvent) -> int:
    """Rough token estimate for one event. Heuristic: len/4 + 1."""
    if isinstance(event, MessageAdded):
        text_len = sum(len(p.text) for p in event.parts if isinstance(p, TextPart))
        return text_len // 4 + 1
    elif isinstance(event, ToolCallRecorded):
        return (len(event.tool_name) + len(event.input_json)) // 4 + 1
    elif isinstance(event, ToolResultRecorded):
        return (len(event.tool_call_id) + len(event.output_text)) // 4 + 1
    return 0

def estimate_session_tokens(session: Session) -> int:
    """Sum token estimates across all session events."""
    return sum(_estimate_event_tokens(e) for e in session.events)
```

### should_compact

```python
def should_compact(
    session: Session,
    config: CompactionConfig,
    existing_summary_sections: list[str] | None = None,
) -> bool:
    """True when the session exceeds the compaction budget.

    Requires BOTH: enough messages to preserve AND enough tokens.

    `existing_summary_sections` is accepted for symmetry with `compact_session`
    but is NOT counted against the budget -- the summary lives in
    `system_prompt`, not in `session.events`, so it can't re-trigger
    compaction of itself.
    """
    messages = derive_messages(session)

    if len(messages) <= config.preserve_recent_messages:
        return False

    token_sum = sum(_estimate_message_tokens(m) for m in messages)
    return token_sum >= config.max_estimated_tokens
```

### compact_session (main function)

```python
def compact_session(
    session: Session,
    config: CompactionConfig,
    existing_summary_sections: list[str] | None = None,
) -> CompactionResult:
    """Compact a session by summarizing older messages and preserving the recent tail.

    Algorithm:
    1. If should_compact is False, return CompactionResult unchanged.
    2. Derive messages. Use `existing_summary_sections` (engine-owned) to
       know whether we are re-compacting.
    3. Calculate keep_from boundary (messages.len - preserve_recent_messages).
    4. Walk boundary backward to avoid splitting tool-use/tool-result pairs.
    5. Build structured summary of removed messages.
    6. If re-compacting, merge previous summary highlights.
    7. Build `summary_sections` list (preamble + summary + recent-note + resume).
    8. Rebuild Session from preserved messages + paired tool events. NO
       synthetic system event is inserted.
    9. Return CompactionResult.
    """
    existing_summary_text = _existing_summary_text(existing_summary_sections)

    if not should_compact(session, config, existing_summary_sections):
        return CompactionResult(
            summary_sections=existing_summary_sections or [],
            formatted_summary=format_compact_summary(existing_summary_text or ""),
            compacted_session=session,
            removed_message_count=0,
        )

    messages = derive_messages(session)
    raw_keep_from = max(0, len(messages) - config.preserve_recent_messages)
    keep_from = _safe_boundary(messages, raw_keep_from, prefix_len=0)

    removed = messages[:keep_from]
    preserved = messages[keep_from:]

    summary = _merge_compact_summaries(
        existing_summary_text,
        _summarize_messages(removed),
    )
    formatted_summary = format_compact_summary(summary)
    summary_sections = build_summary_sections(
        summary,
        recent_messages_preserved=len(preserved) > 0,
    )

    # Build new Session from preserved messages only. No synthetic system event.
    compacted_session = _rebuild_session(session, preserved)

    return CompactionResult(
        summary_sections=summary_sections,
        formatted_summary=formatted_summary,
        compacted_session=compacted_session,
        removed_message_count=len(removed),
    )


def build_summary_sections(
    summary: str,
    recent_messages_preserved: bool,
) -> list[str]:
    """Wrap a structured summary into ordered system_prompt sections.

    The engine prepends these sections to the static system_prompt before
    every `run_turn`. Each section is a separate list element so Anthropic's
    prefix cache can match across re-compactions that share a preamble.
    """
    sections: list[str] = [
        COMPACT_CONTINUATION_PREAMBLE,
        format_compact_summary(summary),
    ]
    if recent_messages_preserved:
        sections.append(COMPACT_RECENT_MESSAGES_NOTE)
    sections.append(COMPACT_DIRECT_RESUME_INSTRUCTION)
    return sections


def _existing_summary_text(sections: list[str] | None) -> str | None:
    """Recover the raw summary body from previously-emitted system_prompt sections.

    `build_summary_sections` places the formatted summary at index 1 (after
    the preamble). This inverse reads it back so `_merge_compact_summaries`
    has the same view it would have when re-compacting in-place.
    Returns None if `sections` is falsy or malformed.
    """
    if not sections or len(sections) < 2:
        return None
    return sections[1]  # the formatted summary body


def _rebuild_session(
    original: Session,
    preserved_messages: list[Message],
) -> Session:
    """Create a new Session with the same metadata but only preserved events.

    Walks `original.events` in order and copies:
      - The `SessionStarted` header (session_id, workspace_root, model)
      - Every `MessageAdded` event whose `message_id` appears in
        `preserved_messages`
      - Every `ToolCallRecorded` / `ToolResultRecorded` event linked (by
        `turn_id`) to a preserved ASSISTANT message's turn. This is required
        because `format_session_for_api` rebuilds tool_use / tool_result
        content blocks from these events; dropping them would orphan the
        assistant's tool_use in the preserved message stream.

    Preserves `session_id`, `created_at`, `workspace_root`, `game_name`,
    `model_hint`, and `metadata` on the rebuilt Session. Does NOT insert any
    synthetic system event -- the compacted summary lives in `system_prompt`,
    not in `messages`.
    """
    preserved_message_ids = {m.message_id for m in preserved_messages}

    # Collect turn_ids whose preserved messages contain assistant tool activity.
    preserved_turn_ids: set[str] = set()
    for event in original.events:
        if (
            isinstance(event, MessageAdded)
            and event.message_id in preserved_message_ids
            and event.role == MessageRole.ASSISTANT
        ):
            preserved_turn_ids.add(event.turn_id)

    new_events: list[SessionEvent] = []
    for event in original.events:
        if isinstance(event, SessionStarted):
            new_events.append(event)
            continue
        if isinstance(event, MessageAdded):
            if event.message_id in preserved_message_ids:
                new_events.append(event)
            continue
        if isinstance(event, (ToolCallRecorded, ToolResultRecorded)):
            if event.turn_id in preserved_turn_ids:
                new_events.append(event)
            continue
        # TurnStarted / TurnEnded for preserved turns are also worth copying
        # so replay is consistent. Include if turn_id is preserved.
        if hasattr(event, "turn_id") and event.turn_id in preserved_turn_ids:
            new_events.append(event)

    return Session(
        session_id=original.session_id,
        created_at=original.created_at,
        workspace_root=original.workspace_root,
        game_name=original.game_name,
        model_hint=original.model_hint,
        events=tuple(new_events),
        metadata=original.metadata,
    )
```

### Tool-use / tool-result boundary protection

```python
def _safe_boundary(
    messages: list[Message], raw_keep_from: int, prefix_len: int
) -> int:
    """Walk the compaction boundary backward to avoid splitting tool pairs.

    If the first preserved message is a ToolResult (role=TOOL or first part
    is a ToolResultPart), and the preceding message is an assistant with a
    ToolUse, include the assistant message too. Repeat until the boundary
    is at a safe split point.

    Without this fix, the API rejects orphaned tool results with a 400.
    """
    k = raw_keep_from
    while k > prefix_len:
        first_preserved = messages[k]
        # Check if first preserved message starts with a tool result
        if not _starts_with_tool_result(first_preserved):
            break
        # Check if preceding message has tool use
        preceding = messages[k - 1]
        if _has_tool_use(preceding):
            k -= 1  # include the assistant tool-use message
            break
        k -= 1  # walk back further
    return k
```

### Structured summary builder

```python
def _summarize_messages(messages: list[Message]) -> str:
    """Build a structured summary of removed messages.

    Format follows compact.rs:
    <summary>
    Conversation summary:
    - Scope: N earlier messages compacted (user=X, assistant=Y, tool=Z).
    - Tools mentioned: tool_a, tool_b.
    - Recent user requests:
      - truncated text...
    - Pending work:
      - items containing todo/next/remaining...
    - Current work: most recent non-empty text
    - Key timeline:
      - role: truncated content | tool_use name(input) | tool_result name: output
    </summary>
    """
    user_count = sum(1 for m in messages if m.role == MessageRole.USER)
    assistant_count = sum(1 for m in messages if m.role == MessageRole.ASSISTANT)
    tool_count = sum(1 for m in messages if m.role == MessageRole.TOOL)

    tool_names = sorted(set(_collect_tool_names(messages)))
    recent_user = _collect_recent_role_summaries(messages, MessageRole.USER, limit=3)
    pending = _infer_pending_work(messages)
    current = _infer_current_work(messages)

    lines = [
        "<summary>",
        "Conversation summary:",
        f"- Scope: {len(messages)} earlier messages compacted "
        f"(user={user_count}, assistant={assistant_count}, tool={tool_count}).",
    ]

    if tool_names:
        lines.append(f"- Tools mentioned: {', '.join(tool_names)}.")

    if recent_user:
        lines.append("- Recent user requests:")
        lines.extend(f"  - {r}" for r in recent_user)

    if pending:
        lines.append("- Pending work:")
        lines.extend(f"  - {p}" for p in pending)

    if current:
        lines.append(f"- Current work: {current}")

    lines.append("- Key timeline:")
    for msg in messages:
        role = msg.role.value
        content = _summarize_message_content(msg)
        lines.append(f"  - {role}: {content}")

    lines.append("</summary>")
    return "\n".join(lines)
```

### Re-compaction: merging summaries

```python
def _merge_compact_summaries(existing: str | None, new_summary: str) -> str:
    """Merge a previous compacted summary with a new one.

    If no existing summary, return the new one.
    Otherwise, extract highlights from both and combine into:
    - Previously compacted context: (highlights from old summary)
    - Newly compacted context: (highlights from new summary)
    - Key timeline: (timeline from new summary only)
    """
    if existing is None:
        return new_summary

    prev_highlights = _extract_summary_highlights(existing)
    new_formatted = format_compact_summary(new_summary)
    new_highlights = _extract_summary_highlights(new_formatted)
    new_timeline = _extract_summary_timeline(new_formatted)

    lines = ["<summary>", "Conversation summary:"]

    if prev_highlights:
        lines.append("- Previously compacted context:")
        lines.extend(f"  {h}" for h in prev_highlights)

    if new_highlights:
        lines.append("- Newly compacted context:")
        lines.extend(f"  {h}" for h in new_highlights)

    if new_timeline:
        lines.append("- Key timeline:")
        lines.extend(f"  {t}" for t in new_timeline)

    lines.append("</summary>")
    return "\n".join(lines)
```

### Summary formatting and tag handling

```python
def format_compact_summary(summary: str) -> str:
    """Normalize a compaction summary into human-readable text.

    1. Strip <analysis>...</analysis> blocks entirely.
    2. Replace <summary>content</summary> with "Summary:\ncontent".
    3. Collapse consecutive blank lines.
    """
    without_analysis = _strip_tag_block(summary, "analysis")
    content = _extract_tag_block(without_analysis, "summary")
    if content is not None:
        formatted = without_analysis.replace(
            f"<summary>{content}</summary>",
            f"Summary:\n{content.strip()}",
        )
    else:
        formatted = without_analysis
    return _collapse_blank_lines(formatted).strip()

def get_compact_continuation_message(
    summary: str,
    suppress_follow_up_questions: bool = True,
    recent_messages_preserved: bool = True,
) -> str:
    """Debug helper: return the concatenated continuation text.

    Used by tests / CLI logs that want a single human-readable string.
    Production code uses `build_summary_sections` instead and lets the
    Anthropic SDK join sections with cache breakpoints.
    """
    sections = build_summary_sections(summary, recent_messages_preserved)
    text = "\n\n".join(sections)
    if not suppress_follow_up_questions:
        text = text.replace("\n\n" + COMPACT_DIRECT_RESUME_INSTRUCTION, "")
    return text
```

### Helper functions

```python
def _starts_with_tool_result(message: Message) -> bool:
    """True if the message's first content part is a ToolResultPart."""
    if not message.parts:
        return False
    return isinstance(message.parts[0], ToolResultPart)

def _has_tool_use(message: Message) -> bool:
    """True if the message contains any ToolCallPart."""
    return any(isinstance(p, ToolCallPart) for p in message.parts)

def _collect_tool_names(messages: list[Message]) -> list[str]:
    """Extract all unique tool names from messages."""
    names = []
    for msg in messages:
        for part in msg.parts:
            if isinstance(part, ToolCallPart):
                names.append(part.tool_name)
            elif isinstance(part, ToolResultPart):
                names.append(part.tool_name)
    return names

def _collect_recent_role_summaries(
    messages: list[Message], role: MessageRole, limit: int
) -> list[str]:
    """Last N text snippets from messages of the given role (chronological)."""
    role_texts = [
        _first_text(m) for m in messages
        if m.role == role and _first_text(m)
    ]
    return [_truncate(t, 160) for t in role_texts[-limit:]]

def _infer_pending_work(messages: list[Message]) -> list[str]:
    """Find messages referencing pending/future work."""
    keywords = {"todo", "next", "pending", "follow up", "remaining"}
    results = []
    for msg in reversed(messages):
        text = _first_text(msg)
        if text and any(kw in text.lower() for kw in keywords):
            results.append(_truncate(text, 160))
        if len(results) >= 3:
            break
    return list(reversed(results))

def _infer_current_work(messages: list[Message]) -> str | None:
    """Most recent non-empty text from any message."""
    for msg in reversed(messages):
        text = _first_text(msg)
        if text and text.strip():
            return _truncate(text, 200)
    return None

def _summarize_message_content(message: Message) -> str:
    """One-line summary of a message's content blocks for the timeline."""
    parts = []
    for part in message.parts:
        if isinstance(part, TextPart):
            parts.append(_truncate(part.text, 160))
        elif isinstance(part, ToolCallPart):
            parts.append(f"tool_use {part.tool_name}({_truncate(part.input_json, 80)})")
        elif isinstance(part, ToolResultPart):
            error_prefix = "error " if part.is_error else ""
            parts.append(f"tool_result {part.tool_name}: {error_prefix}{_truncate(part.output_text, 80)}")
    return " | ".join(parts)

def _first_text(message: Message) -> str | None:
    """First non-empty TextPart text, or None."""
    for part in message.parts:
        if isinstance(part, TextPart) and part.text.strip():
            return part.text
    return None

def _truncate(content: str, max_chars: int) -> str:
    """Truncate with ellipsis if over max_chars."""
    if len(content) <= max_chars:
        return content
    return content[:max_chars] + "..."

def _extract_tag_block(content: str, tag: str) -> str | None:
    """Extract text between <tag>...</tag>."""
    start = f"<{tag}>"
    end = f"</{tag}>"
    s = content.find(start)
    if s == -1:
        return None
    s += len(start)
    e = content.find(end, s)
    if e == -1:
        return None
    return content[s:e]

def _strip_tag_block(content: str, tag: str) -> str:
    """Remove <tag>...</tag> block entirely."""
    start = f"<{tag}>"
    end = f"</{tag}>"
    s = content.find(start)
    e = content.find(end)
    if s == -1 or e == -1:
        return content
    return content[:s] + content[e + len(end):]

def _collapse_blank_lines(content: str) -> str:
    """Collapse consecutive blank lines into one."""
    result = []
    last_blank = False
    for line in content.splitlines():
        is_blank = not line.strip()
        if is_blank and last_blank:
            continue
        result.append(line)
        last_blank = is_blank
    return "\n".join(result)

def _extract_summary_highlights(summary: str) -> list[str]:
    """Extract non-timeline highlight lines from a formatted summary."""
    lines = []
    in_timeline = False
    for line in format_compact_summary(summary).splitlines():
        trimmed = line.rstrip()
        if not trimmed or trimmed in ("Summary:", "Conversation summary:"):
            continue
        if trimmed == "- Key timeline:":
            in_timeline = True
            continue
        if in_timeline:
            continue
        lines.append(trimmed)
    return lines

def _extract_summary_timeline(summary: str) -> list[str]:
    """Extract timeline lines from a formatted summary."""
    lines = []
    in_timeline = False
    for line in format_compact_summary(summary).splitlines():
        trimmed = line.rstrip()
        if trimmed == "- Key timeline:":
            in_timeline = True
            continue
        if not in_timeline:
            continue
        if not trimmed:
            break
        lines.append(trimmed)
    return lines
```

## Layer 7 -- `engine.py`: Game Engine (MDP Step Function)

### LLM contract (IG1, IG2)

Two explicit conventions the engine relies on and the LLM must honor:

**Opening generation (IG1).** `generate_opening()` calls
`runtime.run_turn(user_input="Set the scene and deliver your opening line.")`
with a system_prompt whose `turn_instructions` section is built for the
`OPENING` phase. The LLM MUST call `character_respond` (dialogue set,
`mood_intensity=0`, `security_intensity=0`, `invite_turn=False`) AND
`generate_options` (first 3 player options). If `character_respond` is
absent, the engine raises `EngineError`.

**Player-response format (IG2).** `step()` passes the **raw selected option
text** as `user_input` to `run_turn`. No JSON wrapping. No prefix like
"Player chose:". The LLM sees a natural conversational turn and responds
in character.

### Prerequisite: `GameConfig` additions

Amend `realtalk/config.py` `GameConfig` (owned by Layer 2) to add randomized
starting ranges used by `GameEngine.select_role`:

```python
@chz.chz
class GameConfig:
    ...
    starting_mood_range: tuple[int, int] = (30, 50)
    starting_security_range: tuple[int, int] = (40, 60)
```

### What it owns

- `StateSnapshot` -- frozen, serializable game state for RL logging
- `Action` -- the player's composite choice (reaction + response selection)
- `StepResult` -- `(next_state, reward, done, info)`
- `GamePhase` enum -- `ONBOARDING`, `PLAYING`, `ARC_ACTIVE`, `INVITATION_TURN`, `GAME_OVER`
- `GameEngine` -- orchestrates one full game from start to finish
- `GameEngine.step(action) -> StepResult` -- one MDP step
- `snapshot_state(game_state, last_dialogue, phase) -> StateSnapshot` -- mutable->frozen bridge
- `_extract_dialogue_and_options(summary) -> tuple[str, tuple[str, ...]]` -- pulls dialogue + options out of TurnSummary

### MuZero Alignment

The game engine is designed so that a MuZero agent could replace the human player. The engine doesn't know or care whether actions come from a human or a neural network.

```
                    +-----------+
   observation ---> |  Repr. h  | ---> latent state s
                    +-----------+
                          |
                    +-----------+
   s + action ----> |  Dyn. g   | ---> s', reward
                    +-----------+
                          |
                    +-----------+
   s ------------> |  Pred. f  | ---> policy, value
                    +-----------+
```

**Representation (h):** `StateSnapshot` IS the latent state. For a future neural net, this would be embedded. For now, it's the raw state tuple.

**Dynamics (g):** `GameEngine.step()` IS the dynamics function. Given state + action, it produces next_state + reward. The LLM's response is the stochastic part of the transition.

**Prediction (f):** Not built in v2.0. A future RL agent would learn this. For now, the human IS the policy.

```python
@dataclass(frozen=True)
class StateSnapshot:
    """Frozen game state at a point in time. The MDP observation."""
    mood: int
    security: int
    turn_number: int
    arc_active: bool
    arc_turn: int | None
    game_phase: str
    options: tuple[str, ...]  # the 3 response options currently available
    last_dialogue: str        # character's last spoken line

@dataclass(frozen=True)
class Action:
    """Player's composite action for one turn."""
    reaction_direction: str   # "a" or "r"
    reaction_intensity: int   # 1, 2, or 3
    response_choice: int      # 0, 1, or 2 (index into options)

@dataclass(frozen=True)
class StepResult:
    """Result of one MDP step. Matches Gymnasium API shape."""
    state: StateSnapshot      # next state after the action
    reward: float             # immediate scalar reward
    done: bool                # True if game is over
    info: dict[str, object]   # diagnostic data (dialogue, deltas, etc.)

class GamePhase(Enum):
    SETUP_SCENE = "setup_scene"
    SETUP_ROLE = "setup_role"
    OPENING = "opening"
    PLAYING = "playing"
    ARC_ACTIVE = "arc_active"
    INVITATION_TURN = "invitation_turn"
    GAME_OVER = "game_over"
```

### Reward Function

The reward signal must capture: "did the player make the relationship better?"

```python
def compute_reward(
    prev_state: StateSnapshot,
    next_state: StateSnapshot,
    done: bool,
    game_result: str | None,
) -> float:
    """
    Immediate reward for one step.

    Components:
      - Mood delta:     (next.mood - prev.mood) / 100      [-1.0, +1.0]
      - Security delta: (next.security - prev.security) / 100  [-1.0, +1.0]
      - Win bonus:      +5.0 if game_result == "win"
      - Lose penalty:   -3.0 if game_result == "lose"
      - Survival:       +0.05 per turn survived (encourages longer games)
    """
```

The reward is intentionally simple and decomposable. MuZero learns the dynamics model. complex reward shaping would mask the true signal.

### GameEngine

```python
class GameEngine:
    """Orchestrates one game. Wraps ConversationRuntime + ToolRegistry.

    The engine owns:
    - Scene and role (selected during setup)
    - GameState (mutated by tool handlers)
    - ConversationRuntime (for LLM calls)
    - SystemPromptBuilder (rebuilt each step with current state)
    - Compaction (checked after each step)

    The engine does NOT own rendering. It emits state snapshots and
    dialogue text. The TUI reads these.
    """

    def __init__(
        self,
        api_client: ApiClient,
        config: RuntimeConfig,
    ) -> None:
        """
        Wires up:
          - self._config, self._api_client
          - self._game_state: GameState            (set in select_role)
          - self._scene, self._role                (set in select_scene / select_role)
          - self._tool_registry: ToolRegistry      (constructed in select_role once
                                                     game_state exists)
          - self._runtime: ConversationRuntime     (constructed in select_role)
          - self._prompt_builder: SystemPromptBuilder
          - self._compaction_config: CompactionConfig
          - self._compacted_summary_sections: list[str] = []
          - self._trajectory: list[tuple[StateSnapshot, Action, float, StateSnapshot]] = []
          - self._last_dialogue: str = ""
          - self._session_store: SessionStore      (always-on JSONL; D7)
          - self._stored_session: StoredSession
          - self._stored_event_count: int = 0      (high-water mark for incremental append)
        """

    # -- Setup phase --

    def available_scenes(self) -> list[Scene]:
        """Return 3 randomly selected scenes from SCENES."""

    def select_scene(self, scene_id: str) -> None:
        """Lock in the scene. Advances phase to SETUP_ROLE."""

    def available_roles(self) -> list[Role]:
        """Return all 5 ROLES."""

    def select_role(self, role_id: str) -> None:
        """Lock in the role. Initializes GameState with randomized starting values.

        ```python
        import random
        mood = random.randint(*self._config.game.starting_mood_range)
        security = random.randint(*self._config.game.starting_security_range)
        self._game_state = GameState.new(mood, security)
        self._tool_registry = ToolRegistry(
            game_state=self._game_state,
            config=self._config,
            session_id=self._session.session_id,
        )
        self._runtime = ConversationRuntime(
            api_client=self._api_client,
            tool_executor=self._tool_registry,
            session=self._session,
            system_prompt=[],  # replaced per-turn via set_system_prompt
            tool_definitions=self._tool_registry.tool_definitions(),
            on_text=self._on_text,
            model=self._config.game.model,
            max_tokens=self._config.game.max_tokens,
            temperature=self._config.game.temperature,
        )
        # Advance to OPENING.
        ```
        """

    def generate_opening(self) -> str:
        """Ask the LLM to generate the situation description + opening line.

        Contract:
          1. Build system_prompt via `self._prompt_builder.build(scene, role, game_state)`.
          2. Prepend `self._compacted_summary_sections` (empty on fresh game).
          3. `self._runtime.set_system_prompt(prompt)`.
          4. Call `summary = self._runtime.run_turn(
                  "Set the scene and deliver your opening line.")`.
          5. The LLM is expected to call `character_respond` (dialogue,
             mood_intensity=0, security_intensity=0, invite_turn=False) and
             `generate_options` (3 initial options).
          6. `dialogue, options = _extract_dialogue_and_options(summary)`.
          7. If `dialogue == ""`, raise `EngineError("opening: LLM did not call character_respond")`.
          8. Append new events to `self._stored_session`; update high-water mark.
          9. Store `self._last_dialogue = dialogue` and return dialogue.
        """

    # -- Game loop --

    def current_state(self) -> StateSnapshot:
        """Snapshot the current game state."""
        return snapshot_state(self._game_state, self._last_dialogue, self.phase)

    def step(self, action: Action) -> StepResult:
        """Execute one MDP step.

        Full pseudocode (resolves CG4, CG5, CG6, CG7, IG1, IG2, IG4, IG7):

        ```python
        prev_state = self.current_state()

        # 1. Stage 1 reaction (pure game mechanics, never sent to LLM).
        self._tool_registry.execute_reaction(
            direction=action.reaction_direction,
            intensity=action.reaction_intensity,
        )

        # 2. Resolve the chosen option text.
        chosen_option = self._game_state.pending_options[action.response_choice]

        # 3. Rebuild system_prompt with current state + compacted summary.
        base_prompt = self._prompt_builder.build(
            self._scene, self._role, self._game_state,
        )
        prompt = list(self._compacted_summary_sections) + base_prompt
        self._runtime.set_system_prompt(prompt)

        # 4. Run the turn. Raw option text is user_input -- no JSON wrapping, no prefix.
        summary = self._runtime.run_turn(user_input=chosen_option)

        # 5. Persist new events (training-data flywheel; D7).
        new_events = self._runtime.session.events[self._stored_event_count:]
        self._stored_session.append_events(new_events)
        self._stored_event_count = len(self._runtime.session.events)

        # 6. Extract dialogue + options from tool-call inputs.
        dialogue, options = _extract_dialogue_and_options(summary)
        if dialogue:
            self._last_dialogue = dialogue
        # options are also stored in game_state.pending_options by handle_generate_options;
        # we just read them for StateSnapshot.

        # 7. Turn counter + terminal conditions.
        self._game_state.turn_number += 1

        # mood<=0 is already set by handle_character_respond -> game_result="lose".
        # evaluate_choice sets "win" or "lose" on the Invitation Turn.
        # Turn-cap lose check (CG5):
        if (
            self._game_state.game_result is None
            and self._game_state.turn_number >= self._config.game.turn_hard_cap
        ):
            self._game_state.game_result = "lose"

        done = self._game_state.game_result is not None

        # 8. Compaction (between turns only).
        compacted = False
        if should_compact(
            self._runtime.session,
            self._compaction_config,
            self._compacted_summary_sections,
        ):
            result = compact_session(
                self._runtime.session,
                self._compaction_config,
                existing_summary_sections=self._compacted_summary_sections,
            )
            self._runtime.set_session(result.compacted_session)
            self._compacted_summary_sections = result.summary_sections
            compacted = True

        # 9. Build next state, reward, trajectory.
        next_state = self.current_state()
        reward = compute_reward(
            prev_state, next_state,
            done=done,
            game_result=self._game_state.game_result,
        )
        self._trajectory.append((prev_state, action, reward, next_state))

        return StepResult(
            state=next_state,
            reward=reward,
            done=done,
            info={
                "dialogue": dialogue,
                "options": options,
                "compacted": compacted,
                "mood_delta": next_state.mood - prev_state.mood,
                "security_delta": next_state.security - prev_state.security,
                "game_result": self._game_state.game_result,
            },
        )
        ```
        """

    def is_done(self) -> bool:
        """True if game_result is set or turn cap reached."""

    @property
    def phase(self) -> GamePhase:
        """Current game phase."""

    @property
    def game_result(self) -> str | None:
        """'win', 'lose', or None."""

    @property
    def trajectory(self) -> list[tuple[StateSnapshot, Action, float, StateSnapshot]]:
        """Full (s, a, r, s') trajectory for RL logging."""


# --- Bridge helpers (CG6, CG7) ---

class EngineError(RuntimeError):
    """Raised when the engine cannot make progress (e.g., LLM skipped a required tool call)."""


def snapshot_state(
    game_state: GameState,
    last_dialogue: str,
    phase: GamePhase,
) -> StateSnapshot:
    """Bridge mutable GameState -> frozen StateSnapshot."""
    return StateSnapshot(
        mood=game_state.mood,
        security=game_state.security,
        turn_number=game_state.turn_number,
        arc_active=game_state.arc_active,
        arc_turn=game_state.arc_turn,
        game_phase=phase.value,
        options=tuple(game_state.pending_options),
        last_dialogue=last_dialogue,
    )


def _extract_dialogue_and_options(summary: TurnSummary) -> tuple[str, tuple[str, ...]]:
    """Pull dialogue + options out of a TurnSummary's tool-call inputs.

    character_respond.input_json -> {"dialogue": "...", ...}
    generate_options.input_json  -> {"options": ["a", "b", "c"]}

    Returns ("", ()) if neither tool was called -- callers decide whether
    that's an error (generate_opening raises, step() tolerates because the
    LLM might produce a text-only turn in rare cases).
    """
    dialogue, options = "", ()
    for tc in summary.tool_calls:
        try:
            parsed = json.loads(tc.input_json)
        except (json.JSONDecodeError, TypeError):
            continue
        if tc.tool_name == "character_respond":
            dialogue = str(parsed.get("dialogue", ""))
        elif tc.tool_name == "generate_options":
            raw = parsed.get("options", [])
            if isinstance(raw, list):
                options = tuple(str(o) for o in raw)
    return dialogue, options
```

## Layer 8 -- `tui/`: Textual Terminal Interface

### What it owns

- `RealTalkApp` -- the main Textual `App`, manages screen transitions
- `SceneScreen` -- scene selection (3 options)
- `RoleScreen` -- role selection (5 options)
- `SituationScreen` -- displays opening text, press enter to start
- `GameScreen` -- the main game loop screen
- `PostGameScreen` -- results, score, play again
- `StatusBar` widget -- mood/security bars with color + numeric value
- `DialogueArea` widget -- scrolling text area for character dialogue
- `ReactionInput` widget -- captures `a/r` + `1/2/3`
- `OptionPicker` widget -- shows 3 options, numbered selection

### Screen Flow

```
RealTalkApp
    -> SceneScreen      (pick 1 of 3)
    -> RoleScreen        (pick 1 of 5)
    -> SituationScreen   (read opening, press Enter)
    -> GameScreen        (loop until done)
        - Show dialogue
        - Show bars
        - ReactionInput -> OptionPicker -> submit -> LLM turn -> repeat
    -> PostGameScreen    (score, play again?)
        -> back to SceneScreen (if yes)
        -> exit (if no)
```

### GameScreen Layout

```
+----------------------------------------------------------+
|  REALTALK                                    Turn 7 / 25  |
+----------------------------------------------------------+
|                                                          |
|  *She leans back in her chair, arms crossed but          |
|  smiling.*                                               |
|                                                          |
|  "You always pick the window seat. Don't you think       |
|   it gets drafty?"                                       |
|                                                          |
|  MOOD      [===============>     ] 74  (you: +a2)        |
|  SECURITY  [============>        ] 62  (+c1)             |
|                                                          |
+----------------------------------------------------------+
|  How are you feeling? [a/r] + [1/2/3]: _                 |
+----------------------------------------------------------+
```

After reaction input:

```
+----------------------------------------------------------+
|  MOOD      [=================>   ] 81  (you: +a2)        |
|  SECURITY  [============>        ] 62  (+c1)             |
+----------------------------------------------------------+
|  Choose your response:                                   |
|                                                          |
|    1. "I like the draft. Keeps me from getting too       |
|        comfortable."                                     |
|    2. "You've been tracking my seat preferences?         |
|        Should I be flattered?"                           |
|    3. "I can move if it's really bothering you."         |
|                                                          |
|  > _                                                     |
+----------------------------------------------------------+
```

### Key interactions

**Streaming dialogue:** When the LLM responds, text streams into the `DialogueArea` via the `on_text` callback. Each `TextDelta` appends to the display in real time.

**Bar animation:** After the LLM's tool call updates mood/security, the bars animate from the old value to the new value over ~500ms. The delta label (`+a2`, `-r1`) shows what changed.

**Spinner:** While waiting for the LLM, show a spinner in the dialogue area. "Thinking..." with animated dots.

**Graceful exit:** `Ctrl+C` prompts "Quit game? [y/n]" instead of hard-crashing.

### PostGameScreen contract (IG5, D6)

Rendered contents, top-to-bottom:

1. **Result header:** `"YOU WON"` in green on win, `"YOU LOST"` in red on lose.
2. **Final bars:** mood and security with numeric values (re-uses `StatusBar`).
3. **Turn count:** `"Turns played: {engine.trajectory length}"`.
4. **Best moment:** walk `engine.trajectory`, pick the step with the largest
   positive `reward`, and format as:
   `"Best turn: Turn {N} ({action.reaction_direction}{action.reaction_intensity} reaction, mood {prev.mood} -> {next.mood})"`
   where N is `prev.turn_number + 1`. If no positive-reward step exists
   (pure loss), render `"No standout turn."`.
5. **Play again prompt:** `"Play again? [y/n]"`.

Key bindings:
- `y` -> pop back to `SceneScreen` (new game, fresh `GameEngine`).
- `n` or `q` -> `app.exit()`.

### `--no-color` handling (IG6)

Textual reads the `NO_COLOR` env var natively. In `cli.py`:

```python
if "--no-color" in sys.argv:
    os.environ["NO_COLOR"] = "1"
```

No need for a `config.display.no_color` field unless another subsystem
reads it.

### Execution Order

```
Worker A (prompt + compact)  ──────────>  done
                                            |
Worker B (engine)  ─── waits for A ────>  done
                                            |
Worker C (TUI)     ─── waits for B ────>  done
```

Workers A and B can overlap partially: Worker B can build the `StateSnapshot`, `Action`, `StepResult`, `compute_reward`, and `GamePhase` types without prompt.py. Worker B blocks on Worker A only for the `step()` implementation (which needs `SystemPromptBuilder`).

**Alternative parallel strategy:** Worker A builds prompt.py. Worker B builds engine.py types + reward. Worker C builds widgets + screens with mock engine. Then final integration wires them together.

```
Worker A (prompt + compact)  ──────────────────>  done ──┐
Worker B (engine types + reward + step stub)  ────────>  │─> Integration
Worker C (TUI widgets + screens + mock engine)  ──────>  │
                                                         └─> Wire + CLI
```

This lets all three workers start immediately. Integration is a fourth step that wires the real engine into the real TUI.

## What this spec does NOT cover

- **Neural network training.** We structure data for MuZero. We don't train models.
- **MCTS planning.** MuZero uses MCTS to search ahead. Not built here. The human IS the search.
- **Multiplayer / networked play.** Single player, single character.
- **Persistent save/load between sessions.** Each game is fresh (PRD v1 out of scope).
- **bash.py / file_ops.py / mcp.py.** Coding-agent infrastructure. Not needed for the game.
- **Post-game summary scoring algorithm.** The PRD defines components (consistency, attunement, patience, final arc) but v2.0 shows win/lose + final bars + turn count. Detailed scoring is a polish pass.
