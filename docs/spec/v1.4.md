> Layer 3 (`api.py`) delivered the streaming API client and event types.
> **v1.4 builds the conversation runtime — the engine that drives the game loop.**

> **Supersedes v1.3.** The v1.3 spec proposed litellm multi-provider and FastAPI server.
> That plan is retired for v1 — the game uses the Anthropic SDK directly. Multi-provider
> is a v2 concern. v1.4 assumes the `AnthropicClient` in `api.py` is implemented with

> client must emit it.

## Architecture

### 1. The Immutable Session Bridge

The central design challenge: `Session` is frozen. Each `add_user_text()`, `record_tool_call()`, etc. returns a **new** Session. The runtime must track evolving state across multiple API calls within a single turn.

**Solution:** `self._session` is a mutable reference to the latest immutable snapshot.

```python
class ConversationRuntime:
    def __init__(self, ..., session: Session, ...):
        self._session = session  # re-bound on every mutation

    def _update_session(self, new_session: Session) -> None:
        self._session = new_session

    @property
    def session(self) -> Session:
        """The current session snapshot (read-only)."""
        return self._session
```

The caller retrieves the final session via `runtime.session` after `run_turn()` returns. This is the same pattern as Rust's `&mut self` on the runtime — the Session itself is never mutated, but the runtime's reference advances forward.

### 2. Message Format Bridge

The Anthropic Messages API expects:

```python
{"role": "user", "content": "hello"}
{"role": "assistant", "content": [
    {"type": "text", "text": "Let me check..."},
    {"type": "tool_use", "id": "tu_1", "name": "update_mood", "input": {"delta": 5}}
]}
{"role": "user", "content": [
    {"type": "tool_result", "tool_use_id": "tu_1", "content": "mood updated to 65"}
]}
```

The session stores events (`MessageAdded`, `ToolCallRecorded`, `ToolResultRecorded`) as separate entries. A pure function `format_session_for_api()` bridges the two by walking `session.events` directly. See [§ format_session_for_api](#format_session_for_api-the-bridge-function) below.

### 3. The Turn Loop

```
run_turn(user_input: str) -> TurnSummary
  1. Open turn:          session, turn_id = start_turn(session, opened_by="player")
  2. Record user input:  session, msg_id  = add_user_text(session, turn_id, user_input)
  3. INNER LOOP (max_iterations guard):
     a. Format messages:   api_messages = format_session_for_api(session)
     b. Build request:     ApiRequest(system_prompt, api_messages, tools, model, max_tokens)
     c. Stream response:   for event in client.stream(request):
                             TextDelta  → accumulate text, call on_text callback
                             ToolUse    → collect into pending_tool_uses list
                             UsageEvent → accumulate into total_usage
                             MessageStop → break
     d. Record assistant:  session, _ = add_assistant_text(session, turn_id, accumulated_text)
                           for each tool_use in pending_tool_uses:
                             session, call_id = record_tool_call(session, turn_id, tu.name, tu.input)
     e. If no tool uses:   break inner loop
     f. Execute tools:     for each (tool_use, call_id):
                             try:
                               result = executor.execute(tu.name, tu.input)
                               is_error = False
                             except Exception as exc:
                               result = f"ERROR: {type(exc).__name__}: {exc}"
                               is_error = True
                             session, _ = record_tool_result(session, turn_id, call_id, result, is_error)
     g. iterations += 1
     h. Continue inner loop (goto 3a)
  4. Close turn:         session = end_turn(session, turn_id, status)
  5. Return TurnSummary
```

**Iteration safety valve:** `max_iterations` (default 10) prevents infinite tool loops. If exceeded, the turn ends with `TurnStatus.FAILED` and `TurnSummary.hit_iteration_limit = True`.

### ConversationRuntime

```python
class ConversationRuntime:
    """The conversation engine. Drives the turn loop.

    Takes an ApiClient and ToolExecutor as constructor arguments. Neither is
    hardcoded. In tests, inject MockClient/ScriptedClient and NoOpExecutor.
    In production, inject AnthropicClient and the game's tool registry.

    The runtime holds a mutable reference to the current (immutable) Session.
    Each session mutation rebinds self._session to the new snapshot.
    """

    def __init__(
        self,
        api_client: ApiClient,
        tool_executor: ToolExecutor,
        session: Session,
        system_prompt: list[str],
        tool_definitions: list[dict[str, object]],
        on_text: Callable[[str], None],
        model: str = "claude-haiku-4-5-20251001",
        max_tokens: int = 8096,
        temperature: float = 1.0,
        max_iterations: int = 10,
    ) -> None: ...

    @property
    def session(self) -> Session:
        """The current session snapshot (read-only)."""
        ...

    def run_turn(self, user_input: str) -> TurnSummary:
        """Execute one full turn: user input → assistant response(s) → tool execution loop."""
        ...
```

**Constructor parameters:**

| Parameter | Purpose |
|-----------|---------|
| `api_client` | Injected. `MockClient`/`ScriptedClient` in tests, `AnthropicClient` in production |
| `tool_executor` | Injected. `NoOpExecutor`/`EchoExecutor`/`StaticExecutor` in tests, game mechanics in production |
| `session` | Current session snapshot. Runtime rebinds internally on every mutation |
| `system_prompt` | List of prompt sections. Passed through to `ApiRequest` unchanged |
| `tool_definitions` | Anthropic tool definition dicts. Empty list `[]` for no-tool turns |
| `on_text` | Callback fired for each `TextDelta`. `print` in production, capturing lambda in tests |
| `model` | Model identifier string. Caller passes `config.game.model`; no default in production |
| `max_tokens` | Max response tokens per API call |
| `temperature` | Sampling temperature. Caller passes `config.game.temperature` |
| `max_iterations` | Safety valve for the tool loop |

### ScriptedClient (Multi-call MockClient) — lives in `api.py`

The existing `MockClient` yields all events in a single `stream()` call and is exhausted. The tool loop requires multiple sequential `stream()` calls — one per iteration. `ScriptedClient` fills this gap.

**This class belongs in `api.py`**, not `conversation.py`. It is a test double for the `ApiClient` protocol, alongside `MockClient`. The conversation module imports it from there.

```python
class ScriptedClient:
    """Serves different event sequences on successive stream() calls.

    Each stream() call pops the next sequence. Raises IndexError if called
    more times than there are sequences. Records each ApiRequest for assertions.

    >>> client = ScriptedClient([
    ...     [ToolUse("t1", "Bash", '{}'), MessageStop()],
    ...     [TextDelta("done"), MessageStop()],
    ... ])
    >>> list(client.stream(ApiRequest(...)))  # first call: tool use
    >>> list(client.stream(ApiRequest(...)))  # second call: text only
    >>> client.call_count
    2
    """

    def __init__(self, sequences: list[list[AssistantEvent]]) -> None:
        self._sequences = list(sequences)
        self._index = 0
        self.requests: list[ApiRequest] = []

    @property
    def call_count(self) -> int:
        return self._index

    def stream(self, request: ApiRequest) -> Iterator[AssistantEvent]:
        if self._index >= len(self._sequences):
            raise IndexError(
                f"ScriptedClient exhausted: {self._index} calls made, "
                f"only {len(self._sequences)} sequences provided"
            )
        self.requests.append(request)
        events = self._sequences[self._index]
        self._index += 1
        yield from events
```

## `format_session_for_api`: The Bridge Function

```python
def format_session_for_api(session: Session) -> list[dict[str, object]]:
    """Build Anthropic-format messages from the full session event stream.

    Walks session.events directly (not derive_messages) because the API
    format requires interleaving tool_use blocks in assistant messages
    and tool_result blocks in user messages.

    Returns a list of dicts ready for ApiRequest.messages.
    """
```

### Why walk events directly?

`derive_messages()` returns only `MessageAdded`-derived `Message` objects. `ToolCallRecorded` and `ToolResultRecorded` are tracked separately on `SessionView`. Walking `session.events` lets us build the correct interleaving without cross-referencing.

### Event → API message mapping

| Session Event | API Output |
|---------------|------------|
| `SessionStarted` | skip |
| `TurnStarted` | skip |
| `TurnEnded` | skip |
| `FeedbackRecorded` | skip |
| `SessionMetadataUpdated` | skip |
| `MessageAdded(role=USER)` | `{"role": "user", "content": text}` |
| `MessageAdded(role=ASSISTANT)` | Start new assistant message with text block |
| `ToolCallRecorded` | Append `{"type": "tool_use", ...}` to current assistant message |
| `ToolResultRecorded` | Append `{"type": "tool_result", ...}` as user-role message |

### Merging rules

The Anthropic API rejects consecutive messages with the same role. The algorithm:

1. Walk events, converting each to `(api_role, content_blocks)`.
2. If the previous output message has the same `api_role`, append blocks to it.
3. Otherwise, start a new output message.

**Edge cases:**

- **Assistant message with only tool calls (no text):** Omit the empty text block. The assistant message contains only `tool_use` blocks.
- **Multiple tool results:** Two `ToolResultRecorded` events produce two `tool_result` content blocks merged into a single user-role message.
- **Mixed text + tool_use in one response:** A single assistant message with `[{"type": "text", ...}, {"type": "tool_use", ...}]`.

### Content block format

```python
# TextPart → Anthropic text block
{"type": "text", "text": "Let me check..."}

# ToolCallRecorded → Anthropic tool_use block
{"type": "tool_use", "id": "call_abc123", "name": "update_mood", "input": {"delta": 5}}
# Note: input_json is json.loads()'d into a dict

# ToolResultRecorded → Anthropic tool_result block (in a user-role message)
{"type": "tool_result", "tool_use_id": "call_abc123", "content": "mood is now 65"}
# If is_error=True, add "is_error": true
```

## Implementation Details

### Text Accumulation

`TextDelta` events are accumulated into a string buffer during streaming. After `MessageStop`, the accumulated text is recorded as a single `add_assistant_text()` call:

```python
text_buffer: list[str] = []
pending_tool_uses: list[ToolUse] = []

for event in self._api_client.stream(request):
    if isinstance(event, TextDelta):
        text_buffer.append(event.text)
        self._on_text(event.text)
    elif isinstance(event, ToolUse):
        pending_tool_uses.append(event)
    elif isinstance(event, UsageEvent):
        total_input += event.input_tokens
        total_output += event.output_tokens
        total_cache_creation += event.cache_creation_tokens
        total_cache_read += event.cache_read_tokens
    elif isinstance(event, MessageStop):
        break

accumulated_text = "".join(text_buffer)
```

### Error Handling in the Tool Loop

When `executor.execute()` raises, the runtime catches it and records the error as a tool result:

```python
try:
    result_text = self._tool_executor.execute(tool_use.name, tool_use.input)
    is_error = False
except Exception as exc:
    result_text = f"ERROR: {type(exc).__name__}: {exc}"
    is_error = True
```

The model sees the error text and can react. The runtime never crashes due to a tool executor failure.

### Assistant Messages with Tool Calls

When the model responds with both text and tool calls, the session records:
1. One `add_assistant_text()` with the accumulated text (may be empty string `""`).
2. One `record_tool_call()` per `ToolUse` event.

**Design decision:** Every assistant API response produces exactly one `add_assistant_text()` call, even if the text is empty. This keeps the event stream predictable and simplifies `format_session_for_api` (every assistant turn starts with a `MessageAdded` event).

### Turn Lifecycle

`run_turn()` owns the full turn lifecycle:
- `start_turn()` at the top
- `end_turn()` at the bottom with `TurnStatus.COMPLETED` or `TurnStatus.FAILED`

The caller gets back a session with a closed turn. The turn_id is included in `TurnSummary`.

### Empty Stream Handling

If `stream()` yields no events (or only `MessageStop` with no content), record an empty assistant message and end the turn with `TurnStatus.COMPLETED` (the model simply had nothing to say — this is not an error).

## Locked-In Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | `format_session_for_api` walks `session.events`, not `derive_messages()` | `derive_messages()` excludes `ToolCallRecorded` and `ToolResultRecorded`. Walking events directly avoids cross-referencing `SessionView`. |
| D2 | Empty-text assistant messages always recorded | Every assistant API response produces one `add_assistant_text()`, even with `""`. Keeps the event stream predictable. |
| D3 | `run_turn()` calls both `start_turn()` and `end_turn()` | Runtime owns the full turn lifecycle. Caller gets a session with a closed turn. |
| D4 | Executor exceptions → error tool results | `try/except Exception` in the tool loop. Model sees the error text, runtime never crashes. |
| D5 | No permission system in v1.4 | Game doesn't execute arbitrary tools. Permissions are future work. |
| D6 | No auto-compaction in v1.4 | Games are 10–25 turns. Context limits won't be hit. |
| D7 | No hook system in v1.4 | Hooks are a Layer 5 concern. The runtime is clean. |
| D8 | v1.3 litellm/FastAPI plan retired | Multi-provider is premature for a single-character game. Direct Anthropic SDK. v2 concern. |
| D9 | `ScriptedClient` lives in `api.py` | It's a test double for `ApiClient`, same as `MockClient`. Conversation module imports it. |
| D10 | `ApiRequest` gets `temperature` field | Game needs temperature control. Caller passes `config.game.temperature`. |
## Layer 4 — `conversation.py`: The Conversation Loop

**The most important file.** Everything else serves this.

### What it owns

- `ConversationRuntime` — the engine: `run_turn(user_input) -> TurnSummary`
- `ToolExecutor` — Protocol: `execute(tool_name, input) -> str`
- `PermissionPrompter` — Protocol: `decide(tool_name, input) -> bool`
- `TurnSummary` — iterations, usage, tool calls made, compaction info
- Auto-compaction trigger: check token count after every turn

### The turn loop

```
1. Append UserMessage to Session
2. Assemble ApiRequest (system_prompt + full session history + tool definitions)
3. stream() → collect TextDelta (render), ToolUse (queue), UsageEvent (track), MessageStop (stop)
4. Append AssistantMessage (text + tool use blocks) to Session
5. If tool uses:
     for each ToolUse:
       execute via ToolExecutor
       append ToolResult to Session
     goto 2  ← give model the results
6. If no tool uses: turn is done
7. Check token count → maybe compact
8. Return TurnSummary
```

The loop is **re-entrant by design**. The model drives itself: it calls tools until it decides it's done, then emits `MessageStop` with no tool calls. An agent implementing a complex task may loop 20–30 times before stopping.

### Key design decisions

**`ConversationRuntime` takes both `ApiClient` and `ToolExecutor` as constructor arguments.** Neither is hardcoded. In tests, inject a `MockClient` and a `FakeExecutor`. In production, inject `AnthropicClient` and `ToolRegistry`. The runtime is unaware of either implementation.

**Rebuild the runtime each turn in the CLI layer.** Between turns, the session is the only persistent state. The runtime is ephemeral. This ensures config changes, new MCP servers, or CLAUDE.md edits take effect on the next turn without restarting.

**Auto-compaction fires after the turn, not during.** The token count is checked after `MessageStop`. If over the threshold (default 100,000 input tokens, overridable via env var), `compact_session()` is called before returning `TurnSummary`. The user sees a notice; the turn result is unaffected.

**`on_text` callback for rendering.** Pass `print` in production; pass a capturing function in tests. The runtime never writes to stdout directly.

Here's the critical logic in pseudocode:


run_turn(user_input):
    session.push_user_text(user_input)
    iterations = 0

    loop:
        iterations += 1
        if iterations > max_iterations: ERROR

        events = api_client.stream(system_prompt, session.messages)
        assistant_msg = build_assistant_message(events)
        session.push(assistant_msg)

        tool_uses = extract_tool_uses(assistant_msg)
        if tool_uses.is_empty():
            break                           # model is done talking

        for (id, name, input) in tool_uses:
            pre_hook = run_pre_tool_use_hook(name, input)
            permission = check_permission(name, input, pre_hook)

            if permission == Allow:
                output = tool_executor.execute(name, input)
                post_hook = run_post_tool_use_hook(name, input, output)
                session.push(tool_result(id, output))
            else:
                session.push(tool_result(id, denial_reason, is_error=true))

    maybe_auto_compact()    # if input_tokens > 100k, compress old messages
    return TurnSummary
The loop continues as long as the model keeps requesting tools. Each iteration:

Sends the full conversation history to the API
Receives the model's response
If the response contains tool uses, executes them and loops back
If no tool uses, the model is done — break and return
The Key Types That Wire It Together
The ConversationRuntime is generic over two traits:


pub struct ConversationRuntime<C, T>
where
    C: ApiClient,          // anything that can stream() → Vec<AssistantEvent>
    T: ToolExecutor,       // anything that can execute(tool_name, input) → String
In the real CLI, these are filled by:

C = AnthropicRuntimeClient — makes HTTPS calls to the Anthropic API
T = CliToolExecutor — dispatches to bash, file ops, grep, glob, etc.
In tests, they're replaced with mock structs that return canned responses — that's the whole point of the trait abstraction.

The ML Analogy
If you think of this like a training loop:

ML Concept	Claw Code Equivalent
Training loop (for epoch in epochs)	run_repl() — each prompt is an "epoch"
Forward pass	api_client.stream() — send context, get response
Loss computation / gradient	Tool uses extracted from the response
Backward pass / optimizer step	tool_executor.execute() — changes the world
Logging / callbacks	Hook system (PreToolUse, PostToolUse) + SessionTracer
Gradient clipping	max_iterations — prevents runaway loops
Checkpointing	session.persist() after each turn
Learning rate schedule	Auto-compaction (summarize and drop old messages when tokens > 100k)
The inner loop in run_turn is like an inference loop that keeps running until the model signals "I'm done" (no tool uses in its response). Each iteration is one model call + tool execution round.
