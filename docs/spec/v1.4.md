## Layer 4 — `conversation.py`: The Conversation Loop

**The most important file.** Everything else serves this.

### What it owns

- `ConversationRuntime` — the engine: `run_turn(user_input) -> TurnSummary`
- `ToolExecutor` — Protocol: `execute(tool_name, input) -> str`
- `PermissionPrompter` — Protocol: `decide(tool_name, input) -> bool`
- `TurnSummary` — iterations, usage, tool calls made, compaction info
- Auto-compaction trigger: check token count after every turn

### The turn loop

```
1. Append UserMessage to Session
2. Assemble ApiRequest (system_prompt + full session history + tool definitions)
3. stream() → collect TextDelta (render), ToolUse (queue), UsageEvent (track), MessageStop (stop)
4. Append AssistantMessage (text + tool use blocks) to Session
5. If tool uses:
     for each ToolUse:
       execute via ToolExecutor
       append ToolResult to Session
     goto 2  ← give model the results
6. If no tool uses: turn is done
7. Check token count → maybe compact
8. Return TurnSummary
```

The loop is **re-entrant by design**. The model drives itself: it calls tools until it decides it's done, then emits `MessageStop` with no tool calls. An agent implementing a complex task may loop 20–30 times before stopping.

### Key design decisions

**`ConversationRuntime` takes both `ApiClient` and `ToolExecutor` as constructor arguments.** Neither is hardcoded. In tests, inject a `MockClient` and a `FakeExecutor`. In production, inject `AnthropicClient` and `ToolRegistry`. The runtime is unaware of either implementation.

**Rebuild the runtime each turn in the CLI layer.** Between turns, the session is the only persistent state. The runtime is ephemeral. This ensures config changes, new MCP servers, or CLAUDE.md edits take effect on the next turn without restarting.

**Auto-compaction fires after the turn, not during.** The token count is checked after `MessageStop`. If over the threshold (default 100,000 input tokens, overridable via env var), `compact_session()` is called before returning `TurnSummary`. The user sees a notice; the turn result is unaffected.

**`on_text` callback for rendering.** Pass `print` in production; pass a capturing function in tests. The runtime never writes to stdout directly.

### Acceptance criteria

```python
def test_single_turn_no_tools(tmp_path):
    events = [TextDelta("Hello"), MessageStop()]
    runtime = ConversationRuntime(
        api_client=MockClient(events),
        tool_executor=NoOpExecutor(),
        session=Session.new_at(tmp_path / "s.jsonl", "test", tmp_path),
        system_prompt=["You are helpful."],
        tool_definitions=[],
        on_text=lambda *a, **kw: None,
    )
    summary = runtime.run_turn("hi")
    assert summary.iterations == 1
    assert len(runtime.session.messages) == 2  # user + assistant

def test_tool_loop(tmp_path):
    # First call: model calls Bash. Second call: model stops.
    events_turn1 = [ToolUse("t1", "Bash", '{"command":"ls"}'), MessageStop()]

        api_client=MockClient(events_turn1 + events_turn2),
        tool_executor=EchoExecutor(),  # returns input as output
        ...
    )
    summary = runtime.run_turn("list files")
    assert summary.iterations == 2
    assert "Bash" in summary.tool_calls
```

fn run_repl(...) -> Result<()> {
    let mut cli = LiveCli::new(model, true, allowed_tools, permission_mode)?;
    let mut editor = input::LineEditor::new("> ", ...);
    println!("{}", cli.startup_banner());        // prints the ASCII CLAW art

    loop {
        match editor.read_line()? {
            Submit(input) => {
                // 1. check /exit, /quit
                // 2. try SlashCommand::parse → handle_repl_command
                // 3. try bare-word skill dispatch
                // 4. otherwise: cli.run_turn(&trimmed)
            }
            Cancel => {}       // Ctrl-C: swallow, keep looping
            Exit => {          // Ctrl-D: save and exit
                cli.persist_session()?;
                break;
            }
        }
    }
}
The key thing: run_repl never touches the model directly. It delegates every user turn to cli.run_turn(). The REPL's job is input handling (line editor, tab completion, slash commands), not inference.

Here's the critical logic in pseudocode:


run_turn(user_input):
    session.push_user_text(user_input)
    iterations = 0

    loop:
        iterations += 1
        if iterations > max_iterations: ERROR

        events = api_client.stream(system_prompt, session.messages)
        assistant_msg = build_assistant_message(events)
        session.push(assistant_msg)

        tool_uses = extract_tool_uses(assistant_msg)
        if tool_uses.is_empty():
            break                           # model is done talking

        for (id, name, input) in tool_uses:
            pre_hook = run_pre_tool_use_hook(name, input)
            permission = check_permission(name, input, pre_hook)

            if permission == Allow:
                output = tool_executor.execute(name, input)
                post_hook = run_post_tool_use_hook(name, input, output)
                session.push(tool_result(id, output))
            else:
                session.push(tool_result(id, denial_reason, is_error=true))

    maybe_auto_compact()    # if input_tokens > 100k, compress old messages
    return TurnSummary
The loop continues as long as the model keeps requesting tools. Each iteration:

Sends the full conversation history to the API
Receives the model's response
If the response contains tool uses, executes them and loops back
If no tool uses, the model is done — break and return
The Key Types That Wire It Together
The ConversationRuntime is generic over two traits:


pub struct ConversationRuntime<C, T>
where
    C: ApiClient,          // anything that can stream() → Vec<AssistantEvent>
    T: ToolExecutor,       // anything that can execute(tool_name, input) → String
In the real CLI, these are filled by:

C = AnthropicRuntimeClient — makes HTTPS calls to the Anthropic API
T = CliToolExecutor — dispatches to bash, file ops, grep, glob, etc.
In tests, they're replaced with mock structs that return canned responses — that's the whole point of the trait abstraction.

The ML Analogy
If you think of this like a training loop:

ML Concept	Claw Code Equivalent
Training loop (for epoch in epochs)	run_repl() — each prompt is an "epoch"
Forward pass	api_client.stream() — send context, get response
Loss computation / gradient	Tool uses extracted from the response
Backward pass / optimizer step	tool_executor.execute() — changes the world
Logging / callbacks	Hook system (PreToolUse, PostToolUse) + SessionTracer
Gradient clipping	max_iterations — prevents runaway loops
Checkpointing	session.persist() after each turn
Learning rate schedule	Auto-compaction (summarize and drop old messages when tokens > 100k)
The inner loop in run_turn is like an inference loop that keeps running until the model signals "I'm done" (no tool uses in its response). Each iteration is one model call + tool execution round.
