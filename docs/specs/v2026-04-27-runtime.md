## 0. One-line summary

A single FastAPI service that turns an inbound SMS into a 5-minute voice coaching call by orchestrating Twilio (telephony), Pipecat (frame pipeline), Hume EVI (speech-to-speech with prosody), and the Claude Agent SDK (intake/character/feedback brains), persisting every frame to `sessions/{id}/` for eval and training.

## 1. Audience

This spec is for the engineer building the runtime. It assumes familiarity with async Python, FastAPI, and WebSocket plumbing; it does *not* assume prior Pipecat or Hume EVI experience and points to docs where needed.

The spec is component-shaped. Each numbered component in §3 has a contract (inputs, outputs, dependencies, errors). Build them in the order listed in §10. The eval harness is being built in parallel from `v2026-04-27-eval-harness.md`; this runtime must produce session artifacts byte-equivalent in shape to what eval emits.

## 2. End-to-end flow

```
                                                          ┌─────────────┐
                                                          │   User's    │
                                                          │   phone     │

        │     │       │   │                                                │         │
        │     │       │   └── (TranscriptionFrame, ProsodyFrame, audio)    │         │

        │     │  ArtifactWriter sinks (parallel branch)                    │         │

        │  on hangup:                                                                │
        │   SessionOrchestrator.finalize() ─▶ SynthesisWorker (background task)      │
        │                                       ├ StoryAgent  (Claude Agent SDK)     │
        │                                       └ FeedbackAgent (Claude Agent SDK)   │

        │                                     update session.json (complete)         │
        │                                              │                             │
        │                                              ▼                             │

        │                                     "Your rehearsal: <viewer-url>"         │
        │                                                                            │
        │  GET  /viewer/{id}     → web/viewer.html                                   │

        │                                                                            │
        └────────────────────────────────────────────────────────────────────────────┘
```

Three swap points define the system:

1. **Phase boundary** (intake → practice, practice → feedback): `PhaseProcessor` emits a `PersonaSwitchFrame`; `HumeEVIService` swaps its EVI config (different voice + system prompt) without dropping the audio session.
2. **Per-turn LLM brain** (every Hume turn during practice and intake): Hume EVI calls the **CLM webhook** on our service; we route to a coach or character `ClaudeAgent` based on current phase + session context, stream tokens back.
3. **Post-call synthesis**: a background task spawns two Claude Agent SDK runs (story, feedback) with read-only tools over the persisted artifacts. Decoupled from the call; can be replayed on frozen artifacts.

## 3. Components

Each component has: **purpose**, **interface** (Python signature or HTTP contract), **dependencies**, **failure modes**. Components are ordered roughly outside-in.

### C1. Telephony Gateway (`rehearse/telephony.py`)

**Purpose**: own all Twilio interaction — webhook handlers, outbound call placement, SMS send, Media Streams WS attach.

**Interface** (FastAPI routes + a small client):

```python
class TelephonyClient(Protocol):
    async def place_call(self, to: str, callback_url: str) -> str:        # returns call_sid
    async def send_sms(self, to: str, body: str) -> str:                  # returns message_sid

# routes (mounted on the app):
POST /twilio/sms      # inbound SMS — trigger
POST /twilio/voice    # voice webhook — returns TwiML with <Stream>
WS   /media/{session_id}  # Twilio Media Streams duplex
POST /twilio/status   # call status callbacks (ringing, answered, completed, failed)
```

**Dependencies**: `twilio` SDK, env-configured account SID + auth token + sending number.

**Failure modes**:
- Twilio signature validation fails → 403, log, drop.
- Outbound call rejected (busy, no-answer) → mark session `failed`, no SMS sent.
- WS handshake from non-Twilio source → 401, drop.
- WS drops mid-call → pipeline tears down, finalize runs on whatever is captured, session marked `partial`.

**Notes for the engineer**: validate `X-Twilio-Signature` on every webhook. Use `MEDIA_FORMAT_INBOUND_MULAW_8000` for Pipecat's `TwilioFrameSerializer`. The TwiML response from `/twilio/voice` embeds the WS URL with the session_id path segment so the WS handler knows which session is connecting.

### C2. Session Orchestrator (`rehearse/session.py`)

**Purpose**: lifecycle owner — mint a session, hold a handle through the call, finalize on hangup, persist the manifest, kick off synthesis.

**Interface**:

```python
class SessionOrchestrator:
    async def start(self, trigger: TriggerEvent) -> SessionHandle: ...
    async def attach_pipeline(self, session_id: str, ws: WebSocket) -> None: ...
    async def finalize(self, session_id: str, status: CompletionStatus) -> None: ...

class SessionHandle:
    session_id: str
    session_dir: Path
    started_at: datetime
    consent: ConsentState
```

**Dependencies**: `ArtifactStore` (C9), `TelephonyClient` (C1), `SynthesisWorker` (C8), `SMSNotifier` (C1, helper on top of `send_sms`).

**Failure modes**:
- Disk full at session-mint → 503 to webhook, no call placed.
- Pipeline crash mid-call → finalize writes whatever frames landed, status=`partial`.
- Synthesis crash → status=`partial`, SMS sends with a "transcript only" message.

**Notes**: the orchestrator is stateful in-process (a `dict[session_id, SessionHandle]`). v0 single-replica only. Horizontal scale would require external state (see §5 storage trade-offs).

### C3. Pipeline Builder (`rehearse/pipeline.py`)

> **Superseded** by `docs/specs/v2026-04-28-drop-pipecat.md` §4–§5. The owned-runtime design replaces the Pipecat `Pipeline` factory with a per-call `asyncio.TaskGroup` wiring `TwilioStream`, `HumeEVIClient`, `FrameBus`, and writer/processor tasks. The component contracts below describe the original Pipecat-based plan; read alongside the drop-pipecat spec for what's actually being built.

**Purpose**: pure factory — given a `session_id` and a transport, return a configured Pipecat `Pipeline` instance ready to run.

**Interface**:

```python
def build_pipeline(
    session_id: str,
    transport: BaseTransport,        # FastAPIWebsocketTransport for prod, SimulatedTransport for eval
    artifact_store: ArtifactStore,
    config: RuntimeConfig,
) -> Pipeline: ...
```

**Dependencies**: Pipecat 0.0.50+, `HumeEVIService`, `PhaseProcessor`, artifact-writer processors.

**Wiring** (frame flow top-to-bottom inside the pipeline):

```
transport.input()            # AudioRawFrame in
  → HumeEVIService            # speech-to-speech + prosody/transcription events
  → PhaseProcessor            # consumes timer + transcription, emits control frames
  → ParallelTee:
        ├── TranscriptWriter
        ├── ProsodyWriter
        ├── AudioRecorder
        └── TelemetryLogger
  → transport.output()        # AudioRawFrame out
```

The audio path is `transport → Hume → transport`; the writer branch is a parallel observer fed by `PhaseProcessor`'s outbound frames. Pipecat supports this with `ParallelPipeline` / `ProducerProcessor` patterns.

**Failure modes**: configuration errors (missing API keys, bad voice IDs) raise at `build_pipeline`, not mid-call. Audit `RuntimeConfig.validate()` on app startup.

### C4. PhaseProcessor (`rehearse/phases.py`)

**Purpose**: the only stateful custom processor. Owns `Phase` state, drives transitions on wall-clock + soft cues, emits two custom control frames.

**Interface**:

```python
class PhaseProcessor(FrameProcessor):
    def __init__(
        self,
        budgets: dict[Phase, int],           # seconds per phase
        cue_detector: PhaseCueDetector,      # injected; pure function over recent transcript
    ): ...

    async def process_frame(self, frame: Frame, direction: FrameDirection) -> None: ...
```

**Inputs (frames consumed)**: `TickFrame` (Pipecat clock; or we synthesize via `asyncio.timeout`), `TranscriptionFrame`, control frames.

**Outputs (frames emitted)**:
- `PhaseTransitionFrame(from_phase, to_phase, reason)` — boundary marker for writers.
- `PersonaSwitchFrame(target_persona, voice_config_id, system_prompt)` — instructs `HumeEVIService` to swap its active EVI config without closing the audio session.

**Dependencies**: pure — no I/O. The cue detector is a regex/heuristic in v0 ("the user said goodbye", "it's time to wrap up", silence > 8s with explicit acknowledgment) and can be promoted to an LLM call later behind the same protocol.

**Failure modes**: budget overrun > 10% → forces transition with `reason="budget_overrun"`. Cue detector exception → log, continue, fall through to budget-based transition.

**Test contract**: pure frame-in / frame-out (`tests/test_phases.py`). No Hume, no Claude.

### C5. HumeEVIService (Pipecat service, configured by us)

> **Superseded** by `docs/specs/v2026-04-28-drop-pipecat.md` §5.4. `pipecat-ai[hume]` 1.1.0 ships only Hume TTS, not the EVI service this section assumed. The replacement is `rehearse/services/hume_evi.py` — a `HumeEVIClient` wrapping `hume.empathic_voice.AsyncHumeClient`, publishing events to a `FrameBus` rather than emitting Pipecat frames. Event-to-frame mapping table lives in the drop-pipecat spec.

**Purpose**: speech-to-speech voice with prosody as first-class events. We do **not** write this — it ships in `pipecat-ai[hume]`. We configure two EVI configs and handle config swap on `PersonaSwitchFrame`.

**Two configs**:

| Config | Voice | System prompt source | CLM endpoint |
|---|---|---|---|
| `coach` | calm, warm preset (Hume voice ID pinned in env) | `rehearse/personas.py:COACH_PROMPT` (constant) | `POST /hume/clm/{session_id}?role=coach` |
| `character` | neutral preset, overridden per-session | `compile_character(intake)` output, injected at session start | `POST /hume/clm/{session_id}?role=character` |

**Config swap behavior** (Hume EVI specifics — pinned to docs Q1):

> **Open question Q1 from SPEC**: does Hume EVI support hot config swap on a live session, or do we tear down + reopen WS? If teardown is required (~500ms cost), we accept it during the explicit phase boundary — user is being told "let's hear how this would actually go" while the swap happens, hiding the latency. The engineer must verify in spike work before locking the design.

**Failure modes**: Hume WS drop → Pipecat's `HumeEVIService` reconnects per SPEC §4.3 (1 attempt, ≤2s backoff). On second failure, pipeline ends, session marked `partial`.

### C6. Claude CLM Endpoint + Agents (`rehearse/agents/`)

**Purpose**: serve the Hume Custom Language Model webhook with Claude Agent SDK-driven brains. This is where the *thinking* of the coach and the character lives.

**HTTP contract** (Hume EVI calls this on every turn):

```
POST /hume/clm/{session_id}?role={coach|character}
Content-Type: application/json
Body: Hume CLM request — conversation history + last user message + emotion vector
Response: Server-sent events streaming text tokens, terminated per Hume CLM spec
```

**Agent dispatch**:

```python
class ClaudeAgent(Protocol):
    async def respond(
        self,
        session_id: str,
        history: list[CLMTurn],
        last_user_emotion: ProsodyScores | None,
    ) -> AsyncIterator[str]: ...

# concrete agents:
class CoachAgent(ClaudeAgent): ...        # intake elicitation + feedback delivery
class CharacterAgent(ClaudeAgent): ...    # plays the compiled CounterpartyPersona
```

**Why Claude Agent SDK over raw Anthropic SDK**:

- The intake agent is naturally an *agent*: it has tools (`record_intake_field(name, value)`, `mark_intake_complete()`), runs a multi-turn elicitation, and we want structured output landing in `IntakeRecord` without parsing a freeform transcript.
- The character agent benefits less from tools but the SDK's session/history management is cleaner than rolling our own.
- The feedback agent (post-call, C8) heavily uses tools (`read_transcript`, `read_prosody`, `find_incongruence(turn_id)`, `cite_turn(utterance_id)`). The SDK is the natural shape.

Single SDK across all three keeps the dependency footprint small and the prompt-engineering surface uniform.

**Per-agent tool surface** (v0):

| Agent | Tools | System prompt source |
|---|---|---|
| `CoachAgent` (intake mode) | `record_intake_field`, `mark_intake_complete` | `personas.COACH_INTAKE_PROMPT` |
| `CoachAgent` (feedback mode) | `read_transcript`, `read_prosody`, `find_incongruence`, `cite_turn` | `personas.COACH_FEEDBACK_PROMPT` |
| `CharacterAgent` | none — pure dialogue | compiled `CounterpartyPersona.personality_prompt` |
| `StoryAgent` (post-call, C8) | `read_intake`, `read_transcript` | `personas.STORY_PROMPT` |
| `FeedbackAgent` (post-call, C8) | same as `CoachAgent (feedback)` plus `write_feedback_md` | `personas.FEEDBACK_PROMPT` |

The intake/practice agents run inside the call (latency-critical, streamed); the story/feedback agents run post-call (correctness-critical, batched).

**Failure modes**: Claude API timeout → fallback line ("hold on, let me think") streamed to Hume so the user isn't left in silence; structured retry. Tool-call schema violation → log, swallow, continue dialogue. Rate-limit → 503 to Hume, which the engineer must handle by configuring Hume's CLM retry policy (Hume docs Q3).

**Open question Q3**: Hume CLM SSE format — does Anthropic's streaming output map cleanly, or do we need a translator? The engineer spikes this on day 1.

### C7. Artifact Writers (`rehearse/writers/`)

> **Partially superseded** by `docs/specs/v2026-04-28-drop-pipecat.md` §4. Writers stay one-per-artifact, but each is now a plain async task that consumes a `FrameBus.subscribe()` queue rather than a Pipecat `FrameProcessor`. The artifact contracts (filenames, line formats, failure semantics) below are unchanged.

**Purpose**: parallel sinks that observe pipeline frames and persist them to disk in real time. One processor per artifact.

**Interface** (uniform, all four):

```python
class ArtifactWriter(FrameProcessor):
    def __init__(self, session_id: str, store: ArtifactStore): ...
    # internal: append-on-frame; close-on-EndFrame.
```

| Writer | Consumes | Produces |
|---|---|---|
| `TranscriptWriter` | `TranscriptionFrame` | `transcript.jsonl` (one `TranscriptFrame` per line) |
| `ProsodyWriter` | `ProsodyEventFrame` (Hume-emitted) | `prosody.jsonl` (one `ProsodyFrame` per line) |
| `AudioRecorder` | `AudioRawFrame` (both directions) | `audio.wav` (16kHz PCM, both channels mixed) |
| `TelemetryLogger` | `LLMUsageFrame`, `TTSAudioRawFrame` start/end markers | `telemetry.jsonl` (`InferenceLogEntry` + `LatencyBreakdown`) |

**Dependencies**: `ArtifactStore` (C9). All writes go through the store, never direct filesystem calls — this is the seam that makes the storage choice swappable (§5).

**Failure modes**: write error → log, continue (don't kill the call). Session manifest marks any artifact whose writer errored as missing; orchestrator status becomes `partial`.

### C8. Synthesis Worker (`rehearse/synthesis.py`)

**Purpose**: post-call. Two pure(-ish) functions over a frozen session, each implemented as a Claude Agent SDK run.

**Interface**:

```python
async def synthesize_story(session: Session, store: ArtifactStore) -> Path: ...     # → story.md
async def synthesize_feedback(session: Session, store: ArtifactStore) -> Path: ...  # → feedback.md
```

**Inputs**: only the persisted session artifacts (intake, transcript, prosody). Both are deterministic-modulo-temperature; both can be re-run on a frozen session for replay or A/B.

**Run policy**: `SessionOrchestrator.finalize()` enqueues a background `asyncio.Task` that runs both synthesis functions in parallel, then writes `feedback.md` + `story.md` + updates `session.json` (`completion_status="complete"`), then triggers `SMSNotifier`.

**Budget**: SPEC §4.1 — call end → SMS in ≤90s. Both synthesis runs together must fit in ~75s to leave headroom for SMS round-trip.

**Failure modes**: synthesis timeout → write a stub `feedback.md` ("Your rehearsal was captured. Synthesis failed; transcript and audio available."), status=`partial`, SMS still sends.

**Test contract**: replayable on a frozen session fixture. The eval harness uses these same functions in `target=synthesis`.

### C9. ArtifactStore (`rehearse/storage.py`)

**Purpose**: storage abstraction. One interface, multiple backends. Every component that persists session data goes through this.

**Interface**:

```python
class ArtifactStore(Protocol):
    def session_dir(self, session_id: str) -> Path: ...
    async def write(self, session_id: str, name: str, data: bytes | str) -> None: ...
    async def append(self, session_id: str, name: str, line: str) -> None: ...   # for jsonl
    async def open_audio(self, session_id: str) -> AudioWriteHandle: ...           # streaming wav
    async def read(self, session_id: str, name: str) -> bytes: ...
    async def list_sessions(self) -> AsyncIterator[str]: ...
    def public_url(self, session_id: str, name: str) -> str: ...                   # for viewer SMS
```

**v0 implementation**: `LocalFilesystemStore` writing to `sessions/{session_id}/`. `public_url` returns a URL on the same FastAPI app's static handler.

Storage trade-offs are §5.

### C10. Artifact Viewer (`web/viewer.html` + `/viewer/{id}` route)

**Purpose**: public read-only HTML view of a session — embedded transcript, audio player, prosody chart, feedback. Linked from the post-call SMS.

**Interface**: `GET /viewer/{session_id}` returns `web/viewer.html` (static) which fetches `/sessions/{id}/*` files via JSON. No auth in v0; URLs carry an unguessable session_id (uuid4 hex, 128 bits of entropy). See §6 security.

**Failure modes**: missing artifacts → page shows "session was incomplete; here's what we captured."

## 4. Entry point

Single FastAPI app. One process, one event loop, all components mounted on it.

```python
# rehearse/app.py
from fastapi import FastAPI
from rehearse.config import RuntimeConfig
from rehearse.session import SessionOrchestrator
from rehearse.storage import LocalFilesystemStore
from rehearse.telephony import mount_twilio_routes
from rehearse.agents.clm import mount_clm_routes
from rehearse.agents.dispatch import AgentRegistry

def create_app(config: RuntimeConfig | None = None) -> FastAPI:
    config = config or RuntimeConfig.from_env()
    app = FastAPI(title="rehearse")

    store = LocalFilesystemStore(root=config.session_root)
    agents = AgentRegistry.default(config)
    orchestrator = SessionOrchestrator(store=store, agents=agents, config=config)

    mount_twilio_routes(app, orchestrator, config)
    mount_clm_routes(app, agents)
    app.mount("/sessions", StaticFiles(directory=config.session_root))
    app.mount("/viewer", StaticFiles(directory="web", html=True))

    return app

app = create_app()

# run: uvicorn rehearse.app:app --host 0.0.0.0 --port 8080
```

`RuntimeConfig.from_env()` validates: Twilio creds, Hume API key + two EVI config IDs (coach + character template), Anthropic API key, public base URL (for Twilio webhooks), session root path. Missing or malformed → fail fast at startup.

## 5. Storage choices and trade-offs

Storage is the most consequential infrastructure decision because it touches the call's hot path (artifact writes during the call), the post-call synthesis path, and the viewer URL. Three real options:

### 5.1 Option A — Local filesystem only (recommended for v0)

`sessions/{session_id}/` on the same disk as the FastAPI process. `public_url` serves via FastAPI's `StaticFiles`.

| | |
|---|---|
| ✅ | Zero infrastructure. No bucket setup, no DB, no IAM. |
| ✅ | Fastest writes — local fsync, no network round-trip during the call. Critical because writers run on every frame. |
| ✅ | Easiest debugging — `ls sessions/{id}/`, `cat transcript.jsonl`. |
| ✅ | Eval harness reads from the same path with no abstraction. |
| ❌ | Single-replica only. If the FastAPI pod restarts during a call, the session is partial and stays on the dead node's disk. |
| ❌ | No durability. A disk failure loses every session. |
| ❌ | Viewer URL is the same hostname as the live service — coupling availability concerns. |
| ❌ | Doesn't scale past one box. |

**When this fails us**: as soon as we want HA, multi-replica deploys, or independent uptime for the viewer. Realistic horizon: 4–6 weeks once non-founder users start using it.

### 5.2 Option B — Local filesystem + S3 mirror on finalize

Same as A during the call. On `SessionOrchestrator.finalize()`, the entire session directory is uploaded to an S3 (or GCS) bucket; `public_url` returns the bucket URL (CloudFront/Cloud-CDN-fronted, signed). Local files retained for N days then garbage-collected.

| | |
|---|---|
| ✅ | Live-call write path stays fast (local). |
| ✅ | Durable post-call (S3 11 9s). |
| ✅ | Viewer scales independently — bucket + CDN, not the FastAPI service. |
| ✅ | Replay/debug from S3 if local was lost. |
| ❌ | Adds an upload step inside the 90s call-end → SMS budget. ~10MB / 5min call uploads in 1–3s on Cloud Run egress, fits the budget but eats headroom. |
| ❌ | Adds infra: bucket, IAM role, CDN. Real but small. |
| ❌ | Eval harness reads paths that are now sometimes-local-sometimes-S3 — store interface absorbs this but a test must cover both. |

**Recommendation**: this is the natural step 2. Build the `ArtifactStore` interface in v0 (Option A backend) so the swap to A+S3 mirror is a single new backend class, not a refactor.

### 5.3 Option C — Postgres index + object store from day one

Session manifest (`Session` row) lives in Postgres; artifact files (`*.jsonl`, `*.wav`, `*.md`) in S3. The viewer queries Postgres for the manifest, then signed-URLs the files.

| | |
|---|---|
| ✅ | Queryable. "All sessions with consent=granted last week" is `SELECT`, not `find \| jq`. |
| ✅ | Multi-replica safe immediately — Postgres handles the lock state, not in-process dicts. |
| ✅ | Real schema migrations via Alembic — early discipline. |
| ❌ | Postgres on the live-call write path (writing the manifest at consent, at phase transition, at finalize) — small writes but real network hops. ~5–20ms each on a healthy connection; adds up across phase transitions. |
| ❌ | Schema duplication: pydantic types in `types.py` *and* SQL tables. Either ORM-mapping (SQLAlchemy + pydantic adapter) or hand-syncing. SPEC §14 "single schema everywhere" gets harder. |
| ❌ | Operational overhead from day one — backups, migrations, connection pooling. Heavy for a prototype. |
| ❌ | Couples eval to a database — eval harness now needs Postgres or a fixture layer. |

**Verdict**: probably the right end-state at scale; wrong for v0. Defer until session volume + query needs justify it (likely after live deployments to >10 invited users).

### 5.4 Decision

**v0 ships Option A.** The `ArtifactStore` interface is shaped now so the migration to B (and later C) is a backend swap, not a refactor. The PR that adds Option B will land before non-founder users come online.

Concrete migration triggers, written down so they're not someone's vibe later:
- Move to **B** when: any of (a) non-founder user invited; (b) Cloud Run replica count > 1; (c) we want viewer URLs that survive a deploy.
- Move to **C** when: any of (a) sessions/week > 200; (b) we want cross-session queries for analytics; (c) we add user accounts.

## 6. Cross-cutting concerns

### 6.1 Concurrency model

One asyncio event loop, one process. Each call is one task tree:

```
SessionOrchestrator (top-level task)
 ├─ Pipecat Pipeline (async generator over WS frames)
 │   ├─ Hume EVI WS (outbound, owned by HumeEVIService)
 │   ├─ Artifact writers (each a FrameProcessor task)
 │   └─ PhaseProcessor + clock task
 └─ on hangup → SynthesisWorker task (Claude Agent SDK runs)
```

Capacity is bound by Hume + Anthropic rate limits, not by our process. No CPU-bound work on the hot path (audio is passed through, prosody comes from Hume); FFI and Pipecat handle the real-time bits. v0 single-replica is fine for ≤ 5 concurrent calls.

### 6.2 Latency budget enforcement

Per SPEC §4.1, the load-bearing number is **800ms p50 user-utterance-end → first AI audio**. The runtime contributes:

- Hume STT → CLM webhook fire: ~50–150ms (Hume internal)
- CLM → Claude first token: ~200–500ms (Claude Sonnet 4.6 typical)
- Claude streaming → Hume TTS first audio: ~100–300ms
- Hume TTS → Twilio: ~50–100ms

We're at the edge of budget. The mitigations available in code:
- `temperature=0.7` and `max_tokens` capped on the practice character to keep responses snappy.
- Stream tokens to Hume as they arrive (no buffering).
- Pre-warm the Claude SDK client at app startup (TLS, connection pool).
- Pin the `CharacterAgent` to Claude Sonnet 4.6, not Opus (Opus is for offline feedback only).

`TelemetryLogger` records `LatencyBreakdown` per turn so we can see where budget goes.

### 6.3 Consent gate (SPEC §8.2)

First Phase-1 utterance from the coach is the consent line. The `CoachAgent` system prompt mandates it; the orchestrator does not advance the session out of "pending" consent until the agent's `record_intake_field("consent", "granted")` tool call lands. If the user declines, the agent says goodbye and the orchestrator finalizes with status=`declined`, deletes audio, retains only the metadata.

### 6.4 Security

- **Session URL entropy**: `session_id` is `uuid4().hex` (128 bits). Treated as a bearer token for the viewer in v0. Acceptable for ≤ a few hundred sessions; revisit when we have user accounts.
- **Twilio webhook signature**: validated on every webhook (C1).
- **Hume CLM webhook**: authenticated via shared secret in header, rotated per env. Hume calls this on every turn — verify before doing any work.
- **Secrets**: env-only. Never committed. `.env.example` documents required keys.
- **PII**: raw transcripts contain names, locations, personal context. Stored encrypted-at-rest (rely on the disk/bucket layer in v0); training-corpus pipeline (out of scope here) does name/location redaction before anything trains.

### 6.5 Observability

- `structlog` JSONL to stdout — picked up by Cloud Run / wherever this deploys.
- Every log line carries `session_id` as the correlation ID.
- `telemetry.jsonl` per session is the in-artifact log, the source of truth for latency and model accounting.
- v0 dashboard is `summary.md` from eval runs (separate spec) plus per-session `telemetry.jsonl`. No Prometheus until volume justifies (per SPEC §11.4).

## 9. Tests

The runtime build is verified at three layers:

| Layer | Where | Asserts |
|---|---|---|
| Pure components | `tests/test_phases.py`, `tests/test_personas.py`, `tests/test_synthesis.py` | Frame-in/out (PhaseProcessor); persona compile (deterministic); synthesis on frozen fixture |

The eval harness (separate spec) is the *quality* gate; this layer is the *correctness* gate.

## 10. Build phasing

Each phase is a PR. Each ends with green tests + a working manual demo.

**Phase R1 — Skeleton + Telephony.**
- C1 Telephony Gateway (Twilio webhook validation, SMS in/out, Media Streams WS attach to a no-op handler).
- C2 SessionOrchestrator (mints sessions, writes session.json, no pipeline yet).
- C9 LocalFilesystemStore.
- Manual demo: SMS triggers a call that connects, says a hard-coded "hello" via Twilio `<Say>`, hangs up. session.json on disk.

**Phase R2 — Pipecat + Hume EVI scaffold.**
- C3 Pipeline Builder with C5 HumeEVIService and trivial pass-through processors.
- Single Hume config (coach), no swap, no phases.
- Manual demo: SMS → call → user talks to coach Hume voice with no agent brain (Hume's default LLM). Audio captured.

**Phase R3 — Phase machine + writers.**
- C4 PhaseProcessor.
- C7 ArtifactWriters (transcript, prosody, audio, telemetry).
- Hard-coded phase boundaries on timer; no soft cues yet.
- Manual demo: 3 phases run in sequence; all four artifact files populated; `telemetry.jsonl` shows latency.

**Phase R4 — Claude CLM + agents.**
- C6 CLM endpoint + `CoachAgent` (intake mode) + `CharacterAgent`.
- Hume points at our CLM endpoint for both configs.
- `compile_character(intake)` becomes real (`personas.py`).
- Manual demo: call where the coach actually elicits structured intake, swaps to a compiled character, runs practice.

**Phase R5 — Post-call synthesis + viewer.**
- C8 SynthesisWorker (`StoryAgent`, `FeedbackAgent` via Claude Agent SDK).
- C10 viewer page wired to served artifacts.
- SMS notification with viewer URL.
- Manual demo: full SMS-to-SMS round-trip; clicking the link shows the rendered session.

**Phase R6 — Reliability + polish.**
- Soft-cue phase transitions in `PhaseProcessor`.
- Hume reconnect path verified.
- Consent gate hardened.
- Twilio signature validation enforced.

**Phase R7 — Storage option B (S3 mirror).**
- New `S3MirrorStore` backend; backend selected by env var.
- Migration playbook in this spec's appendix.
